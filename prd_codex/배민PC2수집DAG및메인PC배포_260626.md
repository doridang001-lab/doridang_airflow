# 배민 PC2 수집 DAG 및 메인PC 배포(ingest)

## Task
배민 수집을 두 번째 컴퓨터(PC2)에서도 돌려 60계정을 분산 수집한다. 현재 `collect_all`은 결과를 OneDrive `analytics/baemin_macro/` 파티션에 직접 업로드하는데, 두 컴퓨터가 같은 OneDrive 파일을 동시에 쓰면 동기화 충돌(`파일-PC2.parquet` 중복본)이 난다. 따라서 PC2는 OneDrive analytics에 직접 쓰지 않고 수집 전체(5종)를 `Collect_Data/영업관리부_수집/_baemin_pc2_inbox/<run>/`에 떨구고, 메인PC의 `DB_UnifiedSales_Dags`/`DB_CollectionCompare_Dags`가 인박스를 읽어 analytics 파티션으로 병합(배포)한다. PC2=하위 절반 계정, 메인PC=상위 절반 → store 파티션 disjoint.

    ```python
        inbox = COLLECT_DB / "영업관리부_수집" / "_baemin_pc2_inbox"
        inbox.mkdir(parents=True, exist_ok=True)
        ts = pendulum.now(KST).format("YYYYMMDD_HHmmss")
        safe = _safe_run_id_part(run_id)
        tmp_run = inbox / f"_tmp_{safe}_{ts}"
        final_run = inbox / f"{safe}_{ts}"
        dst = tmp_run / "baemin_macro"
        shutil.copytree(str(local_baemin), str(dst), dirs_exist_ok=True)
        # manifest: 상대경로 -> size
        manifest = {}
        for p in dst.rglob("*"):
            if p.is_file():
                manifest[str(p.relative_to(tmp_run)).replace("\\", "/")] = p.stat().st_size
        (tmp_run / "manifest.json").write_text(
            json.dumps(manifest, ensure_ascii=False, indent=2), encoding="utf-8"
        )
        tmp_run.rename(final_run)  # 거의-원자적 노출
        return final_run
    ```

  6. (옵션) `_processing` 및 manifest 없는 미완성 폴더 중 N일 경과분 정리.
  7. 반환: 처리 run 수 + dataset별 병합 건수 요약 문자열.

```python
def _save_orders_csv(rows, brand, store, target_date):
    ym = target_date[:7]
    stem = BAEMIN_ORDERS_DB / f"brand={brand}" / f"store={store}" / f"ym={ym}" / f"orders_{ym}"
    new_df = pd.DataFrame(rows, columns=_COLUMNS).astype(str)
    new_order_ids = set(new_df["주문번호"].unique())
    existing = read_table(stem)
    if existing is not None:
        existing = existing[~existing["주문번호"].isin(new_order_ids)]
        combined = pd.concat([existing, new_df], ignore_index=True)
    else:
        combined = new_df
    return write_table(combined, stem)
```

## Constraints
- **PC2는 OneDrive analytics(`ANALYTICS_DB`)에 절대 write 금지.** staging 패치 안에서만 수집하고, 결과는 inbox로만 export.
- **PC2에서 `write_runtime_metrics` 호출 제거** — `DASHBOARD_DB/db_beamin_macro_*_metrics.json` 고정 파일명이라 두 PC가 동시에 쓰면 OneDrive 충돌.
- **PC2에서 `validate_toorder`/`validate_ad_funnel`/`trigger_retry_if_needed`/`precheck_manual_baemin_orders` 제거** — staging 패치 밖에서 OneDrive analytics/toorder에 write하거나 메인PC 전용 DAG 트리거.
- **CSV 데이터셋에 `write_table` 사용 금지** — write_table은 parquet 변환 + 레거시 csv 삭제. now/우가클/shop_operation/monthly_operation/ad_funnel은 csv 유지(`to_csv(index=False, encoding="utf-8-sig")`).
- **ingest는 멱등** — manifest size 검증 + mtime 30분 게이트 + claim-by-move + 성공 후 폴더 삭제. 두 소비자 DAG(메인PC 단일머신)에 둬도 race 없음.
- **포맷/병합키 보존** — orders=주문번호 replace, now=date, ad_funnel=target_date, shop_operation=수집일시, shop_change=전열 dedup keep last, 우가클/monthly_operation=스냅샷 교체.
- PC2 store와 메인 store는 disjoint(하위/상위)지만, 같은 PC2 store가 날짜별 누적되므로 date 계열은 반드시 키 upsert.
- monthly_operation 병합키는 구현 시 `DB_Beamin_monthly_operation`의 `_save`(monthly_operation.csv, line~202)를 직접 읽어 스냅샷/누적 여부 확정.
