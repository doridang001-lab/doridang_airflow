# 출근길 AI 브리핑 3대 결함 수정

## Task
출근길 AI 브리핑(`Private_MorningBriefing_Dags`)이 2026-07-14 아침에 3가지 결함을 동시에 냈다: (1) DB_UnifiedSales가 실제로는 정상인데 `실패 run: failed task 0개`라는 **가짜 FAIL**이 헤드라인에 뜸, (2) 우선순위 LLM이 프리오리티 대신 "추가 정보를 제공해 주실 수 있으신가요?"라고 **되물음**, (3) Flow 캘린더가 **로그인 타임아웃**으로 수집 실패. 라이브 메타DB 조사로 원인을 확정했고(아래), 3개 모두 수정해 브리핑 신뢰도를 올린다.

**조사 결과(확정):**
- 어제(07-13 KST) window에서 DB_UnifiedSales `scheduled` run = success, 자동 `today` run = 25태스크 전부 success → 파이프라인 정상.
- FAIL로 잡힌 14개 run은 전부 codex 수동 백필 run(`codex_unified_after_okpos_recheck14b/14c`, conf에 `sale_date`)이고 각 run은 **24태스크 전부 `skipped` / 실제 실패 태스크 0개**인데 DagRun state만 `failed`. window 내 "failed run 중 진짜 실패 태스크가 있는 run = 0개"를 SQL로 확인.
- 브리핑의 `_collect_previous_day_dag_results*`가 `state='failed'`면 무조건 failures에 넣고, 실패 태스크가 없으니 `실패 run: failed task {0}개` placeholder를 헤드라인으로 띄움.
- 캘린더 실패 원인 = Flow 로그인 타임아웃(this-morning 로그): 로그인 버튼 클릭 후 150초(`FLOW_LOGIN_SUCCESS_TIMEOUT`) 동안 `signin.act`에서 못 벗어남 → `do_login` 1회 시도 실패 → `RuntimeError('Flow 로그인 실패')` → 캘린더 스킵.

## Implementation Steps

### 1. Fix 1 — 가짜 FAIL 제거 (핵심)
`failed` DagRun이라도 **failed/upstream_failed 태스크가 하나도 없으면** failures에서 제외한다. (전부 skipped인 codex 백필 잔여 run = 비액션 아이템)

두 함수 모두 `failed_tasks` 계산 직후에 가드를 넣는다:

- `_collect_previous_day_dag_results` (v1): `failed_tasks = [...]` 계산 직후(약 L996), `run_log_dir = ...` 앞에 추가
```python
if not failed_tasks:
    # failed DagRun이지만 실제 실패 태스크 0개(전부 skipped 등) → 비액션, 스킵
    continue
```
- `_collect_previous_day_dag_results_v2` (약 L1099): `failed_tasks = [...]` 계산 직후 동일 가드 추가.

트레이드오프: 이 가드는 "DagRun은 failed인데 실패 태스크 0개"인 run을 전부 스킵한다. 정상 파이프라인 실패는 항상 failed 태스크 ≥1개이므로 실제 장애는 그대로 잡힌다. `error_summary` placeholder(`실패 run: failed task 0개`) 라인은 자연 소멸. CSV fallback(v1 L1032~)은 건드리지 않는다.

### 3. Fix 3 — Flow 로그인 1회 재시도
`_collect_calendar_events_inner`(약 L118)의 로그인 실패 처리를, **deadline 내 1회 재시도** 후에도 실패하면 기존처럼 raise 하도록 변경:
```python
if not do_login(driver):
    logger.warning("Flow 로그인 1차 실패 — deadline 내 1회 재시도")
    if time.monotonic() >= deadline:
        raise RuntimeError("Flow 로그인 실패")
    time.sleep(3)
    if not do_login(driver):
        logger.warning("Flow 로그인 재시도 실패 — 일정 수집 건너뜀")
        raise RuntimeError("Flow 로그인 실패")
```
`do_login`은 내부에서 `driver.get(LOGIN_URL)`로 다시 진입하므로 재호출만으로 재시도가 된다. 150초×2 ≤ 수집 타임아웃(600초) 안에 든다. 지속 실패 시엔 기존처럼 캘린더만 graceful 스킵(브리핑 나머지는 정상). 반복되면 Flow 로그인 신뢰도 자체가 별도 크롤러 과제이며 이 수정 범위 밖.

LOGIN_URL = "https://flow.team/signin.act"

def do_login(driver) -> bool:
    driver.get(LOGIN_URL)                       # 재호출마다 로그인 페이지 재진입
    # ... id/pw 입력, a#normalLoginButton 클릭 ...
    wait = WebDriverWait(driver, int(os.getenv("FLOW_LOGIN_SUCCESS_TIMEOUT", "150")))
    try:
        wait.until(lambda d: "signin" not in d.current_url.lower()
                   or bool(d.find_elements(By.CSS_SELECTOR, "input#topSearchKeywordBox")))

## Constraints

- Fix 3에서 `do_login` **정의 자체는 수정하지 말 것**(visit_log 크롤러 공유). 브리핑 호출부에서만 재시도.
