# 쿠팡 전체 자동수집 — 경고 제거 / 신뢰도 100% 개선

## Task
64개 매장 쿠팡이츠 크롬확장 배치에서 orders 수집이 15분 runner 타임아웃에 걸려 수집한 주문 전량을 폐기(`0/0`)하고 행이 "경고"로 남는 문제를 없앤다. 근본 원인은 쿠팡의 누적 throttle(주문 상세 패널 silent 지연). **신뢰도 최우선 + 부분 저장 후 백필** 방향으로, content script에 매장별 자체 데드라인을 넣어 부분 데이터를 저장하고, throttle 감지 시 배치를 쿨다운시켜 연쇄를 끊는다.

## Implementation Steps

### 1. (03_coupangeats.js) 매장별 wall-clock 데드라인 도입 — 최우선
- 상수: `ORDERS_STORE_BUDGET_MS = 480000` (8분). 정상 매장(최대 150→27건)도 수 분 내 완료되므로 건강한 매장엔 안 걸림, throttle 매장만 컷.
- `_collectOrdersPages`(아래 Reference `while (!this._stopFlag)` 루프)에 진입 직전 `const _storeStart = Date.now();` 추가. 루프 상단에서
  `if (Date.now() - _storeStart > ORDERS_STORE_BUDGET_MS) { Utils.updateProgressModal({ debug: '⏱ 매장 예산 초과 — 부분 저장 후 백필' }); this._markOrdersTransientError(); hasError = true; break; }`
- `_collectCurrentPageData`의 order 루프(`for (let i...totalItems)`)에도 동일 데드라인 체크를 넣어, 한 페이지가 예산을 넘기면 즉시 빠져나오게 한다(현재 페이지 partial 반영). 데드라인 상태를 리턴에 실어 상위 루프가 저장 후 종료하도록.
- 목적: runner의 15분 외부 타임아웃에 **절대 도달하지 않게** 하여 `_downloadOrdersResult` 호출 기회를 보장.

### 2. (03_coupangeats.js) silent-throttle 조기 bail 가드 수정
- 현재 두 곳(`consecutiveFail >= 5 && capturedCount === 0 && this._getExpectedOrderCount() > 0`)에서 `capturedCount === 0`가 bail을 막는다(throttle는 첫 캡처 이후 시작).
- 변경: `if (consecutiveFail >= 4 && (this._isOrdersThrottleVisible() || this._hadRecentOrdersTransientError()))` → `detailBlocked = true; this._markOrdersTransientError(); break;`
- 기존 `capturedCount===0` 케이스도 유지(둘 중 하나라도 참이면 bail). `_getExpectedOrderCount() > 0` 조건은 유지.

### 3. (03_coupangeats.js) throttle 중 "세부 재수집" 라운드 스킵
- 3-round 재수집 루프(아래 Reference) 진입 전:
  `if (this._isOrdersThrottleVisible() || this._hadRecentOrdersTransientError()) { /* skip re-collection */ }` 로 감싸 활성 throttle이면 재수집(1건 ~25초)을 건너뛴다. 미수집분은 백필로.

### 4. (03_coupangeats.js) 부분 저장 경로 연결
- `_collectOrdersPages`가 `hasError`/`detailBlocked`/데드라인으로 종료돼도 `allRows`를 그대로 상위(`_collectOrdersWithValidation`)로 반환하도록 유지.
- `_collectOrdersWithValidation`에서 오류/제한으로 `return false` 하기 전에 **`allRows.length > 0`이면 `_downloadOrdersResult(allRows, shopInfo, {...stats, isMultiStore})`로 부분 저장 후 `return true`(부분완료 표기)**. 이렇게 하면 CSV가 남고, 미완료 매장은 runner의 백필 큐(미완료 처리)로 익일 재수집된다.
- partial 저장 시 로그: `✅ 저장 완료(부분): {filename}` 로 구분.

### 5. (runner.js) throttle/타임아웃 후 적응형 쿨다운
- 상수 추가: `THROTTLE_COOLDOWN_STEPS = [120000, 240000, 480000]` (2/4/8분), 연속 카운터 `throttleStreak`.
- 계정 루프에서 직전 계정 결과가 `blocked || hardThrottle || op.timeout`(=res.blocked 등)이면 다음 계정 대기시간을 `THROTTLE_COOLDOWN_STEPS[min(throttleStreak++, 2)]` 로 대체하고 `log('↩ 쿨다운 N분 대기...')`. 정상 완료면 `throttleStreak = 0` 로 리셋하고 기존 `rand(DELAY_MIN, DELAY_MAX)` 유지.

### 6. (runner.js) COLLECT_TIMEOUT 하향
- `COLLECT_TIMEOUT = 900000` → `600000`(10분). content script가 8분에 자체 저장하므로 안전망만.

## Test Cases
1. [문법] 확장 로드 오류 없음: `chrome://extensions` 에서 해당 확장 새로고침 → "오류" 배지 없음.
2. [정상 매장] 행 클릭 단일 테스트(주문 있는 매장) → 수 분 내 `완료`, orders CSV 저장(부분 아님).
3. [데드라인] throttle 매장 → 라이브 로그에 `orders 타임아웃 — 0/0개` 대신 `⏱ 매장 예산 초과 — 부분 저장 후 백필` + `✅ 저장 완료(부분): coupangeats_orders_*.csv` 출력.
4. [부분 데이터] 3번 매장의 저장 CSV에 예산 초과 직전까지의 주문 rows가 실제로 존재.
5. [쿨다운] throttle/타임아웃 매장 직후 로그에 `↩ 쿨다운 2분 대기...`(연속 시 4→8분) 삽입, 이후 10057/경고 연쇄 감소.
6. [백필] 배치 종료 로그 `📋 백필 큐 저장 — 미완료 매장 N개`; 익일 배치 시작 시 `미완료 N개 매장 백필 시작`.

## Constraints
- content script는 대형 파일이므로 **지정 함수만 국소 수정**, 전체 재작성 금지.
- 정상 0건(어제 주문 없음) 경로(`_isNormalZeroOrders`)와 hard-throttle(10057) 경로는 기존 동작 유지 — 데드라인/쿨다운이 이들을 오분류하지 않게.
- `ORDERS_STORE_BUDGET_MS(8분) < COLLECT_TIMEOUT(10분)` 부등식 반드시 유지(데드라인이 항상 먼저 발동).
- 부분완료 매장은 runner 백필 큐로 이관되도록 미완료 상태 표기 유지(완료로 마킹 금지).
- OneDrive 개발본만 수정. repo build 폴더는 건드리지 않음.
