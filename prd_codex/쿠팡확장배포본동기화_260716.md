크롬에 로드되는 배포본(`C:\Users\민준\OneDrive - 주식회사 도리당\Extention\doridang_collector_개발용`)이

배포본의 `background.js`/`runner.js`는 구버전 로직(더 과민한 하드차단 판정,
`MAX_AUTO_RETRY=1`)이고, 빌드본에는 사용자가 이미 손본 완화 로직이 있지만 배포본에
반영되지 않았다. `content/03_coupangeats.js`는 반대로 배포본이 더 최신이라 단순
덮어쓰기가 아니라 실제 diff 내용을 비교해 병합해야 한다.

## Implementation Steps

1. **background.js 동기화**
   - `C:\airflow\coupang_extension_build\background.js` 전체를 읽는다.
   - 배포본 `background.js`를 이 내용으로 완전히 교체한다 (배포본은 6/25 구버전으로,
     6/30 빌드본에 추가된 `openRunnerDashboard`, `pendingDownloadNamesByUrl`,
     `sanitizeDownloadFilename` 등 다운로드 파일명/대시보드 로직이 통째로 없다.
     배포본에만 있는 고유 로직 없음 — 안전하게 통째 교체 가능).

2. **runner.js 동기화**
   - `C:\airflow\coupang_extension_build\runner.js` 전체를 읽는다.
   - 배포본 `runner.js`를 이 내용으로 완전히 교체한다.
   - 근거: 두 파일 mtime은 거의 같지만(7/16 09:12 전후) 실제 로직은 다르다.
     배포본은 `MAX_AUTO_RETRY = 1`(재시도 사실상 없음), `isHardThrottlePayload`가
     `payload.blocked`/`row.status === 'blocked'`까지 전부 10057 하드차단으로
     오판하는 구버전 로직이다. 빌드본은 `isRealHardThrottle` 판정을 추가해 "진짜
     10057"과 "일시적 끊김(수집 중 끊김 - 백필 이월)"을 구분하고,
     `partialThrottleStoreNames()`를 신설해 완화했다. 즉 배포본은 사용자가 이미
     고친 로직이 반영되지 않은, 오히려 더 과민한 중간 버전이었다 — "수정했는데
     경고가 더 나온다"는 증상과 정확히 일치. 빌드본이 상위 버전이므로 그대로 채택.

3. **content/03_coupangeats.js 병합**
   - `C:\airflow\coupang_extension_build\content\03_coupangeats.js` (빌드본,
     7/14) 와 배포본 `content/03_coupangeats.js` (7/15) 를 각각 전체 읽는다.
   - 두 파일을 diff하여 (예: `diff -u build/content/03_coupangeats.js deployed/content/03_coupangeats.js`)
     서로에게만 있는 실질적 로직 차이를 식별한다. 공백/BOM/줄바꿈 차이는 무시하고
     실제 코드 의미가 다른 부분만 본다.
   - 알려진 차이 포인트(빌드본 7/14 → 배포본 7/15 사이 배포본에서 추가된 것으로
     보이는 부분): `_storeBranchKey`, `_isCurrentOrdersStore` 최적화 로직.
     빌드본에만 있을 수 있는 것: 주문 합계 검증(order-total validation) 관련 추가
     코드. **두 쪽 다 유지되도록 병합**한다 — 어느 한쪽을 버리지 말 것.
   - 병합된 최종 내용을 하나의 파일 내용으로 만든다.
   - 이 병합본을 (a) `C:\airflow\coupang_extension_build\content\03_coupangeats.js`
     와 (b) 배포본 `content/03_coupangeats.js` 양쪽에 동일하게 반영한다 (빌드본도
     최신 상태로 맞춰서 이후 또 어긋나지 않게 함).

4. **OneDrive 쓰기 권한 처리**
   - 1~3단계에서 배포본 파일 쓰기를 시도할 때 Edit 툴이 `*OneDrive*` 경로에서
     거부되면, PowerShell `Set-Content`/`Copy-Item` 등 파일시스템 직접 쓰기로
     재시도한다.
   - 그것도 거부되면: 병합된 최종 파일 3개를
     `C:\airflow\coupang_extension_build\`에 정리해 두고, 사용자에게 "OneDrive
     폴더로 수동 복사해달라"는 안내(파일 목록 + 대상 경로)를 출력한다. 기존
     `coupang_extension_build/` 안에 있을 수 있는 `COPY_INSTRUCTIONS.md` 패턴을
     참고해 최신화한다.

5. **동기화 후 검증**

     확인한다.
   - `manifest.json`, `content/05_main.js`는 원래 동일했으므로 diff 없음을

## Verification Loop
```
LOOP until all PASS:
  1. Test Cases 1~5 순서대로 실행
  2. FAIL 항목 → 원인 분석 (교체가 누락됐는지, 병합 시 일부 로직이 빠졌는지)
  3. 해당 파일 다시 교체/병합
  4. 전체 Test Cases 재실행
종료 조건: 전체 PASS + 3.의 "양쪽 고유 변경 모두 보존" 조건 위반 없음
```

  git이 `coupang_extension_build/` 쪽을 추적하므로 복구 가능하지만, 배포본은
  git 추적 밖이므로 **교체 전에 배포본 3개 파일을 임시 백업**해 두는 것을 권장)으로
  진행한다.
- 이번 태스크 범위는 "배포본 동기화"로 한정한다. AUTO_LOGIN 재시도 로직 추가,

  동기화로만 한정함).
- git 커밋/푸시는 사용자가 명시적으로 요청하기 전까지 하지 않는다.
