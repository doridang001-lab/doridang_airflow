# 실시간(당일) 매출 추적 — Today 수집·적재 DAG

## Task
현재 매출 파이프라인은 전날 데이터를 새벽~오전에 수집해 오전 9시경 `DB_UnifiedSales`로 최종 적재하므로, "오늘 지금까지 얼마 팔렸는지"는 다음 날 아침에야 알 수 있다.
당일(today) 매출을 오전~오후 여러 번(11/14/17시 KST) 수집→적재하는 별도 "Today" 경로를 추가해 near-real-time 매출 추적을 가능하게 한다.
POS 4종(okpos/unionpos/easypos/posfeed) + 테스트매장 배달수동(배민수동/쿠팡수동)까지 today unified에 반영한다. 기존 전날 확정 파이프라인은 건드리지 않는다.

## Implementation Steps

DB_OKPOS_SALES_TODAY_TIME            = "0 11,14,17 * * *"   # 11:00/14:00/17:00
DB_EASYPOS_SALES_TODAY_TIME          = "5 11,14,17 * * *"   # 11:05/14:05/17:05
DB_UNIONPOS_RECEIPT_TODAY_TIME       = "10 11,14,17 * * *"  # 11:10/14:10/17:10

DB_UNIFIED_SALES_TODAY_DELIVERY_TIME = "20 11,14,17 * * *"  # 배민수동·쿠팡수동 반영 (POS 적재 후)

### 2. POS 소스별 Today DAG (공통 골격, 4개)
각 DAG는 한 run 안에서 수집+적재. `_on_failure_callback`은 `DB_UnifiedSales_Dags.py`의 Telegram+`enqueue_heal_task` 패턴 재사용.
태스크 흐름:
1. **resolve_today** — `today = (conf.get("sale_date") or pendulum.now("Asia/Seoul").format("YYYY-MM-DD"))`, XCom push (key `sale_date`).
2. **collect_today** (`pool="selenium_pool"`) — 기존 소스 수집을 today 날짜로 호출:
   - OKPOS: `download_today_stores`(`DB_OKPOS_Sales.py:1974`) — today 페이지(`PAGE_TYPES["today"]`) 사용, `_mark_no_data`가 today/미래 NO_DATA 스킵.
   - EasyPOS: `_select_yesterday_and_search`(`DB_EasyPOS_Sales.py:1679`)에 `sale_date=today` → days_back=0으로 당일 영수증 수집. **일자별매출 검증(`_upsert_daily_total_csv`, verify_missing)은 today 제외(매출 진행중)이므로 Today DAG에서 호출 금지.**
   - UnionPOS: `_set_date_and_search`(`DB_UnionPOS_Receipt.py:70`)가 `startDate/endDate`를 임의 날짜 DOM 입력 → `sale_date=today` 그대로 동작.
3. **load_today** — `run_<source>(today, overwrite=True)` 호출 → unified 당일 parquet에 source slice 교체 적재(멱등).

**POS Today DAG에서 하지 않는 것:** `validate_sales`, `validate_monthly_sales`, refresh_store_meta 등 마감/검증(당일 부분데이터 오탐). 수집+적재만.

### 3. Posfeed Today — 파이프라인 변경
`DB_Posfeed_Sales.py`의 기본 `download_posfeed_excel`은 날짜피커 미사용 + `등록날짜=yesterday` 하드코딩.
기존 날짜지정 경로 `_click_download(driver, wait, target_date=...)`(`:428`)를 재사용해 `collect_posfeed_today()` 추가:
- `target_date=today`로 `_click_download` → `등록날짜=today` 스탬프 저장(`move_to_storage`의 yesterday 하드코딩 우회).
- 저장 파티션 동일: `posfeed_sales/brand=*/store=*/ym=YYYY-MM/posfeed_orders.csv`.
- 주문상세(`DB_Posfeed_Sales_Detail.py`, 최대 8h)는 Today 범위 제외.
- ⚠ 과소집계: `run_posfeed`의 `_transform_df`가 detail 미매칭 주문을 drop → 상세 없이는 당일 posfeed 매출 과소. v1은 "상세 수집 후 정확"으로 진행(옵션 `keep_unmatched`는 후속 판단).

### 4. 배민수동·쿠팡수동 Today 반영 DAG — `DB_UnifiedSales_Today_Delivery_Dags.py` (필수)
**POS today 적재 뒤** 실행(같은 `unified_sales_<YYMMDD>.parquet` in-place 교정). `TEST_STORES = DELIVERY_MANUAL_TEST_STORES`(`DB_UnifiedSales_common.py:45`, 10개 매장).
태스크(순차):
1. **resolve_today** — POS와 동일.
2. **ingest_manual_baemin** — `load_manual_baemin_orders`(**context) : `COLLECT_DB/영업관리부_수집/baemin_orders_*.csv` → `baemin_macro/orders/…/ym=…`. 반환(로드 건수) XCom push.
3. **load_coupang_macro** — `load_coupang_macro_partition()` : 크롬확장 CSV(`DOWN_DIR` + `영업관리부_수집`의 `coupangeats_orders_*.csv`) → `coupang_macro/orders/…`. 멱등(row hash dedup).
4. **reconcile_baemin (GATE 필수)** — 헬퍼 `_test_stores_with_baemin_today(today)`로 **당일 배민 수동데이터가 있는 매장만** 골라 `reconcile_baemin_for_test_stores(stores=<필터목록>, sale_date=today)` 호출. 필터 결과 비면 전체 스킵.
   - ⚠ 전체 TEST_STORES를 넘기면 수동데이터 없는 매장의 배달의민족 행이 **삭제**된다(함수 docstring 명시). 반드시 gate.
5. **enforce_baemin_manual_only** — `enforce_baemin_manual_only_for_test_stores(stores=<위 필터목록>, sale_date=today)`.
6. **reconcile_coupang** — `reconcile_coupang_for_test_stores(stores=TEST_STORES, sale_date=today)`. 빈 프레임 early-return(삭제 안 함)이라 전체 매장 그대로 전달 안전.
7. **enforce_coupang_manual_only** — `enforce_coupang_manual_only_for_test_stores(stores=TEST_STORES, sale_date=today)`.
8. **cleanup_manual_baemin** — `cleanup_manual_baemin_orders`(**context) : 성공 로드된 CSV만 `_archived`로 이동.

의존성:
`resolve_today >> [ingest_manual_baemin, load_coupang_macro] >> reconcile_baemin >> enforce_baemin_manual_only >> reconcile_coupang >> enforce_coupang_manual_only >> cleanup_manual_baemin`

`_test_stores_with_baemin_today(today)` 명세: `baemin_macro/orders`(`ANALYTICS_DB/baemin_macro/orders/brand=*/store={store}/ym={today[:7]}/orders_*.parquet`)에서 각 test store의 오늘 `주문시각` 행 존재 여부를 확인해, 있는 store만 리스트로 반환. `_parse_baemin_datetime`(`DB_UnifiedSales_baemin.py`) 날짜 파싱 로직 재사용.

### 5. 집계 갱신 DAG — `DB_UnifiedSales_Today_Summary_Dags.py`
`build_daily_summary()`만 호출. 전체 unified parquet 읽어 daily_summary.parquet 재생성 → today slice 반영. `max_active_runs=1`로 단일 writer 보장(4개 소스 DAG 동시 write race 방지).

## Verification Loop
구현 완료 후 아래 루프를 **모든 Test Cases PASS까지** 반복한다.
```
LOOP until all PASS:
  1. Test Cases 순서대로 실행
  2. FAIL 항목 → 원인 분석
  3. 코드 수정
  4. 전체 Test Cases 재실행
종료 조건: 전체 PASS + Constraints 위반 없음
```

## Constraints
- **배민 reconcile는 반드시 gate**: 당일 수동데이터 있는 매장만 대상. 전체 TEST_STORES를 그대로 넘기면 오삭제. (쿠팡은 early-return이라 전체 전달 가능)
- POS Today DAG에서 `validate_sales`/`validate_monthly_sales`/verify_missing 호출 금지(당일 부분데이터 오탐 → Telegram 알림).
- 기존 전날 확정 파이프라인(`DB_UnifiedSales_Dags.py`)·기존 소스 수집 DAG 파일은 **수정 금지**. Today는 신규 파일로만.
- 모든 Today DAG: `dag_id=Path(__file__).stem`, `catchup=False`, `max_active_runs=1`, schedule은 schedule.py 상수 사용, Selenium 태스크 `pool="selenium_pool"`.
- Delivery DAG는 POS Today 적재 뒤(스태거 20분) 실행 — 같은 날짜 parquet 순차 write.
- Summary DAG는 `max_active_runs=1` 단일 writer.
- **데이터 도착 의존성(자동화 불가 영역):** 배민=영업관리부 CSV drop, 쿠팡=크롬확장 수동 실행이 선행돼야 해당 회차에 반영됨. 미도착이면 안전 스킵 후 다음 회차 반영.
