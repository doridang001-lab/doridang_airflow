# OKPOS 전날값 0 버그 수정

## Task
OKPOS DAG가 매일 06:00 KST에 실행되는데, OKPOS 전일 데이터는 당일 12:00 이후에야 조회 가능하다.
그 결과 `okpos_order.csv`에 어제 데이터가 없어 `DB_UnifiedSales` 적재 시 항상 어제가 0이 된다.
추가로, OKPOS daily 페이지가 `'영업일수합: 0 일'` 셸 리포트를 반환할 때 현재 코드가 이를 0으로 영구 저장하여 재시도를 막는 코드 버그도 존재한다.

## Project Conventions
- logging: `logger = logging.getLogger(__name__)` — print 절대 금지
- import 경로: `from modules.transform.utility.XXX import ...` (절대경로)
- 스케줄 상수: `from modules.transform.utility.schedule import ...` (하드코딩 금지)
- cron 표현식: DAG의 timezone이 `Asia/Seoul`이므로 cron은 **KST 기준** 해석됨

## Files to Modify

- `modules/transform/utility/schedule.py` — OKPOS, UnifiedSales 스케줄 시간 변경
- `dags/db/DB_UnifiedSales.py` — OKPOS execution_delta 수정 (2h40m → 2h)
- `modules/transform/pipelines/db/DB_OKPOS_Sales.py` — `'영업일수합: 0 일'` 코드 버그 수정
- `dags/db/DB_OKPOS_Sales_Dags.py` — docstring 스케줄 주석 업데이트

### Step 3: `'영업일수합: 0 일'` 코드 버그 수정 (`modules/transform/pipelines/db/DB_OKPOS_Sales.py`)

`_transform_okpos_daily_df` 함수 내 **케이스 C** (line ~2156) 에서  
`row = row.iloc[:1].copy()` 바로 다음에 추가:

```python
        row = row.iloc[:1].copy()

        # '영업일수합: 0 일' → OKPOS 미마감/매출없음 셸 리포트 → empty 반환하여 NO_DATA 마킹 유도
        if not row.empty:
            days_text = str(row.iloc[0].get("일자", ""))
            if "0 일" in days_text or ": 0 " in days_text:
                logger.info(
                    "daily '영업일수합: 0 일' 감지 → OKPOS 미마감/매출없음으로 판단, empty 반환: %s | %s",
                    sale_date, store_name,
                )
                return pd.DataFrame(), sale_date[:7]
```

**왜 empty를 반환하는가**: `save_to_raw` (line ~2596)에서 `df.empty` 시  
`_mark_no_data()`를 호출 → `_NO_DATA_RETRY_RECENT_DAYS=3` 이내 날짜는 다음 실행에서 재시도.  
0으로 영구 저장되면 `_filter_missing_keys`가 "수집됨"으로 판단해 재시도 안 함.

단, 케이스 C의 세 번째 fallback (`if row.empty and len(data) == 1: row = data.copy()`)에서  
선택된 단일 행에도 같은 체크가 적용되도록 해야 한다. 케이스 C 전체 구조:

```python
# 케이스 C) 단일 일자 리포트
if row.empty:
    row = data[data["일자"].str.startswith("영업일수합:")].copy()
    if row.empty and ("영업매장" in data.columns):
        store_col = data["영업매장"].astype(str).str.strip()
        row = data[store_col.isin(["합계", "총계", "전체"])].copy()
    if row.empty and len(data) == 1:
        row = data.copy()
    if row.empty:
        raise ValueError("daily 엑셀 파싱 실패: '소계:'/'영업일수합:'/합계 행 모두 없음")
    row = row.iloc[:1].copy()

    # ↓ 여기에 추가
    if not row.empty:
        days_text = str(row.iloc[0].get("일자", ""))
        if "0 일" in days_text or ": 0 " in days_text:
            logger.info(
                "daily '영업일수합: 0 일' 감지 → OKPOS 미마감/매출없음으로 판단, empty 반환: %s | %s",
                sale_date, store_name,
            )
            return pd.DataFrame(), sale_date[:7]
```

## Verification Loop

```
LOOP until all PASS:
  1. Test Cases 1~5 순서대로 실행
  2. FAIL 항목 → 원인 분석 (특히 케이스 C에서 row.empty 흐름 추적)
  3. 코드 수정
  4. 전체 Test Cases 재실행
종료 조건: 전체 PASS
```

## Constraints

- `_transform_okpos_daily_df`의 케이스 A, B 로직은 수정하지 않는다
- `'영업일수합: 0 일'` 패턴 체크는 케이스 C 전용 (`row.iloc[:1]` 이후에만 추가)
- schedule.py의 **다른 스케줄 상수는 수정 금지** (OKPOS, UnifiedSales 두 개만 변경)
- `UPSTREAM_POS_TASKS`의 OKPOS 항목만 delta 수정. 나머지 3개(UnionPOS, EasyPOS, Posfeed)는 건드리지 않는다
- Airflow DAG에서 cron은 `tz="Asia/Seoul"` 기준이므로 **UTC 변환 불필요** (KST 값 그대로 사용)
- `_NO_DATA_RETRY_RECENT_DAYS = 3` 상수는 변경 금지 (기존 재시도 로직 유지)
