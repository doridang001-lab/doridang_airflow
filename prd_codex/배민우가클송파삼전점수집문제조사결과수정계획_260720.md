2. **대표 target_date 결정**
   - `read_meta=True`(upload inbox 경로)일 때, 폴더 목록을 순회하기 **전에** 각 폴더의 `_read_meta(folder)`로 target_date를 미리 조회해 `folder → target_date` 매핑을 만든다.
   - `target_date`가 있는 폴더들 중 **최댓값(가장 최신 날짜, 문자열 정렬 `YYYY-MM-DD` 기준)**을 `representative_date`로 정한다. target_date가 하나도 없으면 `representative_date = None`.

3. **폴더별 처리는 기존 로직 유지, meta만 필터링**
   - 기존 58-98행의 폴더별 try/except → 파일 분배 → cleanup 루프는 **그대로 유지**한다(이미 폴더 단위로 격리되어 안전 — 한 폴더 실패가 다른 폴더를 막지 않음).
   - `meta_items.append(folder_meta)` 하는 부분(94-95행)을 아래 로직으로 교체:
     ```python
     if read_meta:
         folder_date = str(folder_meta.get("target_date")) if folder_meta.get("target_date") else None
         if representative_date is None or folder_date == representative_date:
             meta_items.append(folder_meta)
         else:
             logger.warning(
                 "%s 지난 날짜 meta는 downstream 검증에서 제외: folder=%s target_date=%s (대표날짜=%s)",
                 label, folder.name, folder_date, representative_date,
             )
     ```
   - 즉 **파일 분배/cleanup은 대표 날짜와 무관하게 모든 폴더에서 수행**되지만, XCom으로 나가는 `meta`(계정 목록, 검증 결과, 광고매장 등)는 대표 날짜(최신)와 일치하는 폴더의 것만 병합된다.

4. **`_merge_meta` 방어 로직 유지**
   - `_merge_meta`(147-166행)의 `if len(target_dates) > 1: raise ValueError(...)` 방어 로직은 그대로 둔다. 3번 단계에서 이미 같은 날짜의 meta만 들어오도록 필터링했으므로 이 raise는 정상 운영에서는 절대 발동하지 않아야 한다(발동하면 버그이므로 방어선으로 유지하는 게 맞음).

5. **로그 보강**

def ingest(**context) -> str:
    result = ingest_baemin_upload_inbox(**context)
    meta = result.get("meta") or {}
    for key, value in meta.items():
        context["ti"].xcom_push(key=key, value=value)
    return result.get("summary", "upload inbox 적재 완료")

## Constraints
- `_merge_meta`(147-166행)의 다중 target_date 방어(`raise ValueError`)는 삭제하지 말 것 — 필터링 로직이 제대로 동작한다면 이 경로는 도달하지 않아야 하는 안전장치.
- 폴더별 try/except → cleanup 흐름(58-98행)의 구조 자체는 바꾸지 말 것. meta 수집 부분만 조건부로 교체.
- `_baemin_upload_inbox` 실 데이터(OneDrive) 폴더 이동/삭제는 이 작업 범위에 포함하지 않는다 — 코드만 수정한다. 실제 inbox의 stale 폴더 정리는 사람이 별도로 처리.
- 새로운 재시도/폴백 로직, 타임아웃 처리 등 요구하지 않은 기능 추가 금지. 이번 수정 범위는 "meta 병합 필터링"에 한정.
