## Implementation Steps

### B. 재시도 눈덩이 차단 (self-amplification 제거)
1. **재시도 이어받기(resume)**: 현재 `runRetry()`(`runner.js:1306`)/백필은 실패 매장을 통째로 재수집한다.
   `content/03_coupangeats.js` 수집 시 **주문별 수집 상태를 `chrome.storage.local`에 저장**
   (키 예: `ce_collected_orders_{store}_{targetDate}` → 이미 상세패널 파싱 완료한 `order_id` set).
   재수집/재시도 시 이 set에 있는 주문은 **상세패널 펼치기(`btn.click()`, `03_coupangeats.js:1794-1805`)를 건너뛰고**
   저장된 정산값을 재사용 → 상세요청 재발생 0.
2. **10057 = 배치 전체 서킷브레이커**: 현재 `waitBeforeNextAccount`(`runner.js:260`)는 throttle 계정 뒤에만
   2/4/8분 쿨다운을 걸고 **다음 계정으로 계속 진행**한다(같은 IP → 밴 지속).
   → `isHardThrottlePayload`(`runner.js:225`)가 true면 **현재 배치 실행을 일시 중단하고 전체를 30~60분 정지**하는
   `globalCooldownUntil` 타임스탬프를 도입. `runBatch`/`runRetry` 루프 진입부에서 `Date.now() < globalCooldownUntil`이면
   대기. (기존 per-account 쿨다운과 병존, 하드스로틀만 전역 승격.)
3. **재시도 상한 유지**: `MAX_AUTO_RETRY=5`(`runner.js:54`)는 유지하되, 하드스로틀로 중단된 회차는
   재시도 카운트 소모 없이 전역 쿨다운 후 이어가도록 분리.

### C. 부하 분산 (창당 요청 밀도 낮추기)
4. **wave 분할 실행**: 60매장을 여러 wave로 쪼개 wave 사이 긴 간격(예: 60~120분)을 둔다.
   기존 상단/하단 절반 버튼(`runner.js:1178`)을 일반화해 `?wave=N&of=M` 파라미터로 N/M 조각만 실행하도록 확장.
5. **(선택) 상세패널 대상 축소**: `03_coupangeats.js`에서 상세 정산이 꼭 필요한 주문만 펼친다
   (예: 취소/변경 없는 확정 주문 중 이미 저장분은 skip — 1번과 연동). 근본 방식 유지한 채 상세요청 수 자체 감소.

```javascript
function isHardThrottlePayload(payload) {
  const rows = Array.isArray(payload?.storeResults) ? payload.storeResults : [];
  return !!(payload?.blocked || payload?.hardThrottle || rows.some(r => r.blocked || r.hardThrottle || r.status === 'blocked' || String(r.note || '').includes('10057')));
}
function markStoresBlocked(acc, stores, note) { /* 매장 blocked 처리, ordersOk=true → 백필 제외 */ }
function needsThrottleCooldown(res = {}) {
  return !!(res.blocked || res.hardThrottle || res.ordersTimedOut || res.partialThrottle);
}
async function waitBeforeNextAccount(res) {
  let d;
  if (needsThrottleCooldown(res)) {
    d = THROTTLE_COOLDOWN_STEPS[Math.min(throttleStreak, THROTTLE_COOLDOWN_STEPS.length - 1)];
    throttleStreak++;
  } else {
    throttleStreak = 0;
    d = rand(DELAY_MIN, DELAY_MAX);
  }
  await sleep(d);
}
// runBatch 루프: runner.js:1229-1260 / runRetry: 1306 / 백필큐: ce_backfill_queue 1184-1298
```

## Test Cases

1. **[이어받기]** 매장 수집 도중 중단 → 재시도 시, 이미 수집된 order_id는 `btn.click()` 로그가 다시 안 찍히는지
   (콘솔에서 상세패널 펼침 횟수 = 신규 주문 수와 일치) → 기대: 기수집분 재요청 0.

   진행하지 않고 전역 30~60분 대기 로그를 남기는지 → 기대: `globalCooldownUntil` 설정 + 배치 정지.

4. **[회귀]** 정상 경로(throttle 없음)에서 기존 6~12초 계정 간 대기·백필 큐·CSV 다운로드가 그대로 동작하는지 → 기대: 기존과 동일.
