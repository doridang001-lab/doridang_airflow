## 현재 구조 (확인 완료)
- `collect_now_and_woori(account_list, target_date, stability_profile)` (`DB_Beamin_combined.py` L756~) — **계정 순차 루프**(L772 `for account in account_list:`). 계정마다 bootstrap 로그인 → 매장목록 → 매장별 Chrome으로 5단계(now·woori·shop_change·orders·ad_funnel). 매 N매장 Chrome 재기동.

- 매장 루프 본문(L834~919): orders는 `orders_result.get("ok")` False면 `failed_orders` 적재 + `validation_results.extend(...)`. ad_funnel False면 `failed_ads` 적재. 예외는 세션문제(`_is_driver_crash_error`)와 일반 실패로만 분기 — **네트워크 단절 전용 분류 없음**.
- `collect_orders_for_driver` validation 항목: `matched: True|False|None`.
- `beamin_stability.STABILITY_PROFILES` (safe/bulk_70): `initial_stagger_range, account_wait_range, orders_validation_retry, driver_restart_every_stores, max_session_recovery_per_account`. **병렬도 노브 없음.**
- 네트워크 끊김 로그 시그니처: `RemoteDisconnected / Read timed out / Max retries exceeded / Connection refused / HTTPConnectionPool ... timed out / Remote end closed`.

## Implementation Steps

### [2순위] 부분수집 감지 → 자동 재수집

**1. 네트워크 예외 분류 helper (`DB_Beamin_combined.py`)**
`_is_transient_network_error(exc) -> bool` 추가:
```python
_NET_ERR_SIGNS = ("RemoteDisconnected","Read timed out","Max retries","Connection refused",
                  "HTTPConnectionPool","Remote end closed","timed out")
def _is_transient_network_error(exc) -> bool:
    msg = str(exc)
    return any(s in msg for s in _NET_ERR_SIGNS)
```
(`is_driver_crash_error`와 별개 — 크래시 아닌 네트워크 단절.)

**2. 매장 단위 "의심(suspect)" 분류**
- 함수 상단 `suspect_stores: list[dict] = []` 추가.
- 매장 루프 예외 처리(L898~919)에서 `_is_transient_network_error(exc)`이면 `suspect_stores.append({"account": account, "store_info": store_info, "stage": <단계>, "reason": str(exc), "target_date": target_date})`. (기존 failed 적재는 유지.)
- orders validation에서 `matched is False` 또는 `matched is None`인 항목의 매장도 suspect에 포함.
- 반환 dict에 `"suspect": suspect_stores` 키 추가(기존 키 유지).

**3. 자동 재수집 task (`dags/db/DB_Beamin_Macro_Dags.py`)**
- `collect_all`에 `context["ti"].xcom_push(key="suspect", value=result.get("suspect", []))` 추가.
- 신규 함수 `recollect_suspects(**context)`:
  - `suspect = context["ti"].xcom_pull(task_ids="collect_all", key="suspect") or []` — 비면 "재수집 없음" 반환.
  - 안정화 대기(`time.sleep(60)`), 계정별 그룹화 후 `collect_orders_for_account` / `collect_ad_funnel_for_account`를 같은 `target_date`로 1회 호출.
  - 성공/잔존 카운트 요약 문자열 반환.
- PythonOperator `recollect_suspects` (`trigger_rule="all_done"`, `execution_timeout=timedelta(minutes=120)`), 의존성: `t3(retry_failed) >> t_recollect >> [t4,t5,t6]`.

**4. 알림 라인**
- `_build_collection_notification`에 suspect 건수·재수집 잔존 건수 라인 추가(기존 retry/mismatch 라인 옆).

### [3순위] 런타임 최적화

**5. (핵심) 계정 병렬화**
- `beamin_stability.py` 두 프로필에 `max_parallel_accounts` 추가: safe=2, bulk_70=4.
- `collect_now_and_woori`의 계정 루프 본문(L772~ 한 계정 처리 전체)을 `_collect_one_account(account, profile, target_date) -> dict`로 추출. 반환: `{success, fail, failed_accounts, failed_stores, failed_orders, failed_ads, validation, ad_stores, store_info, suspect, metrics_delta}`.
- 메인은 `ThreadPoolExecutor(max_workers=profile["max_parallel_accounts"])`로 계정 제출, 결과를 **메인 스레드에서 병합**(락 최소화). `initial_stagger_range` 지터는 워커 시작 시 유지.
- 계정은 이미 독립(자기 Chrome·자기 매장·store 파티션별 CSV write)이라 계정 간 공유 가변상태 없음. `max_parallel_accounts=1`이면 기존 순차와 동일해야 함.

**6. 고정 sleep → 이벤트 대기**
- `DB_Beamin_04_orders.py`, `DB_Beamin_05_ad_funnel.py`의 명백한 `time.sleep(2.0)` 등을 `WebDriverWait(driver,t).until(조건)`(요소 표시·필터 텍스트 변경·`aria-busy` 해제)으로 치환. React fetch가 DOM 신호와 안 묶인 곳만 짧은 폴링 유지.

**7. (선택) 수집항목 빈도분리**
- `collect_now_and_woori`/`collect_all`에 `collect_stages: list[str] | None`(기본 전체) 추가. conf 또는 요일로 분기: 매일 `["orders","ad_funnel"]`, 주1회(월) `["now","woori","shop_change"]`. 매장당 페이지 이동 5→2.

## Constraints
- 계정은 서로 독립 — 병렬화 시 계정 간 공유 가변상태 금지. 집계는 메인 스레드 병합 또는 `threading.Lock`.
- 반환 dict 기존 키(`summary/failed/validation/ad_stores/store_info_per_account/metrics`) 유지, `suspect`만 추가.
- `collect_*_for_account`/`for_driver` 시그니처·CSV 스키마·`resolve_stability_profile` 시그니처 변경 금지.
- 이미 적용된 나홀로 정규화 수정(`_parse_store_option`/`_filter_store_list_for_request`의 `lookup_store_key`)은 건드리지 말 것.
- `max_parallel_accounts` 기본값 보수적(봇탐지·메모리), `initial_stagger_range` 지터 유지.
- `max_parallel_accounts=1`은 기존 순차 동작과 완전 동일해야 함(회귀 금지).
