# 주문서 교차분석 데이터 마트 (일별 적재)

## Task
"이 메인메뉴를 골랐을 때 어떤 사이드/옵션/다른 메인/음료를 함께 고르는가"를 분석하는 **주문서 교차분석(co-occurrence) 마트**를 일별 자동 적재로 만든다. 소스는 `unified_sales_grp` 일별 parquet(주문 라인 그레인), 분류는 `fin_product_grp_input.csv` 마스터 + `fin_product_map_join.csv` 검수맵 오버레이 + 품목명 fallback이다. 출력은 `order_cross_analysis` 일별 parquet으로, 히트맵/매트릭스 시각화의 소스가 된다.

## Data Facts (설계 근거 — 이미 검증됨)
- 소스: `MART_DB/unified_sales_grp/unified_sales_{YYMMDD}.parquet`. `order_id`가 한 주문의 라인들을 묶음. 하루 ~1만 라인.
- 조인: **`unified.item_id == 마스터 fin_product_grp_input.csv 의 상품코드`** (source별). row 커버리지 98.1%.
- 마스터 제공 컬럼: `is_main_candidate`(Y/N, 98% 채워짐), `중메뉴`(옵션/토핑/사이드메뉴/음료/주류/메인메뉴/점심메뉴/저녁메뉴/단품 메뉴/세트 메뉴 등, `[홀]` 접두어 변형 있음), `대메뉴`, `상품명`, `is_latest`(Y 필터 필요). **주의: 마스터의 `수동분류`/`표준_메뉴명` 컬럼은 전부 NULL이라 못 씀.**
- 검수맵 `fin_product_map_join.csv`(374행): clean `category`(메인/사이드/옵션/토핑/음료/주류/세트/1인/기타) + `standard_menu_name` 제공, 조인키 `item_id`. 커버리지 2%뿐이라 **오버레이 용도로만**.
- menu_name은 okpos만 메인별로 정확. posfeed/배민/쿠팡은 order 전체가 단일 menu_name → 옵션이 어느 메인에 붙는지 라인 단위 불명확. 따라서 그레인은 **order_id(장바구니) 단위 co-occurrence**로 근사한다.
- `is_main_candidate == Y`만 anchor로 쓰면 배달 source 대부분이 누락된다. 명시 메인이 없는 주문은 `menu_name` 기반 synthetic anchor(`MENU::{hash}`)를 생성한다.

## 출력 마트 스키마 (`CROSS_COLUMNS`)
```
sale_date, ym, brand, store,
main_item_id, main_name, main_source,    # anchor 메인(item_id/menu_name)
pair_type,                               # 상대 category: 사이드/옵션/토핑/음료/주류/메인/세트/기타
pair_item_id, pair_name,                 # 상대 품목
co_order_cnt, co_qty, co_amount,         # co_order_cnt = nunique(order_id)
is_multi_main, updated_at                # 멀티메인 과대추정 플래그(bool), 적재시각(KST)
```
unique key: `(sale_date, brand, store, main_item_id, pair_type, pair_item_id)`
저장 경로: `ORDER_CROSS_DIR / f"order_cross_{YYMMDD}.parquet"` (일별 1파일, 하루치 전량 overwrite)

## Implementation Steps
1. **상수 추가**: `paths.py`에 `ORDER_CROSS_DIR = MART_DB / "order_cross_analysis"`, `schedule.py`에 `DB_ORDER_CROSS_ANALYSIS_TIME = "20 9 * * *"`.

2. **모듈 상수** (`DB_OrderCrossAnalysis.py` 상단):
   ```python
   from modules.transform.utility.paths import ORDER_CROSS_DIR, FIN_PRODUCT_MAP_JOIN_CSV_PATH, existing_fin_product_csv_path
   from modules.transform.pipelines.db.DB_UnifiedSales_common import _unified_daily_path, iter_unified_sales_files
   CROSS_ROOT = ORDER_CROSS_DIR
   CROSS_COLUMNS = [...]  # 위 스키마
   ```

3. **분류 로더 `_load_fin_product_category_map() -> dict`**: `FIN_PRODUCT_CSV_PATH`를 `dtype=str`로 로드 → `is_latest=='Y'` 필터 → `상품코드`별 최신 dedup(`updated_at` 있으면 최신, `DB_UnifiedSales_common._load_fin_product_latest` 패턴 복제). 반환: `상품코드 -> {category, is_main, name(상품명)}`. category 파생:
   - `is_main_candidate=='Y'` → `('메인', True)`
   - 아니면 `_normalize_category(중메뉴)` 적용.

4. **`_normalize_category(중메뉴: str) -> str`**: `[홀]`/공백 제거 후 포함검사 — `토핑`→토핑, `사이드`/`곁들임`→사이드, `옵션`/`반반`→옵션, `주류`→주류, `음료`→음료, `세트`→세트, `메인`/`점심`/`저녁`/`단품`→메인, 그 외→기타.

5. **오버레이 로더 `_load_map_join_overlay() -> dict`**: `FIN_PRODUCT_MAP_JOIN_CSV_PATH` 로드 → `item_id -> {category, standard_menu_name}`. non-null만 category/name 덮어씀(검수값 우선).

6. **`_classify_categories(df) -> df`**: sales 라인에 `item_id`(str strip) 기준으로 3의 dict를 map → `category`, `is_main`, `disp_name`(오버레이 standard_menu_name → 마스터 상품명 → item_name fallback) 부착.

7. **`_build_pairs_one_day(date_str) -> pd.DataFrame`**:
   - `_unified_daily_path(date_str)` 로드(없거나 empty면 빈 프레임 반환, hall_viz 방어 패턴).
   - 필수컬럼 방어(`order_id, item_id, qty, total_price, sale_type, brand, store, sale_date`).
   - 필터: `sale_type != '취소'`, `order_id != ''`, `item_id != ''`. `category` 부착 후 `기타` 및 fee-like(배달비/할인/수수료/배달팁/포장비) 제외.
   - `groupby(['source','brand','store','order_id'])`: 각 그룹의 `is_main==True` 라인(mains)과 전체 라인을 order별 **cross merge**(`main_row_id != pair_row_id`, `main_item_id != pair_item_id` 필터) → `pair_type = pair.category`.
   - mains가 없으면 `menu_name`으로 synthetic anchor를 생성하고, 첫 line/main명 동일 line은 pair에서 제외.
   - `is_multi_main` = 해당 order의 고유 메인 item_id 수 >= 2.
   - 방출: `sale_date, ym, brand, store, main_item_id, main_name, pair_type, pair_item_id, pair_name, order_id, pair_qty, pair_amount, is_multi_main`.

8. **`_aggregate_pairs(pairs) -> pd.DataFrame`**:
   ```python
   pairs.groupby(['sale_date','ym','brand','store','main_item_id','main_name','main_source','pair_type','pair_item_id','pair_name']).agg(
       co_order_cnt=('order_id','nunique'), co_qty=('pair_qty','sum'), co_amount=('pair_amount','sum'), is_multi_main=('is_multi_main','any')).reset_index()
   ```
   → `reindex(columns=CROSS_COLUMNS)`, int 캐스팅, `updated_at`(KST) 부여.

9. **`_save_cross_daily(df, date_str) -> int`**: `CROSS_ROOT.mkdir(parents=True, exist_ok=True)` → `_cross_daily_path(date_str)` → 하루치 전량 재계산이므로 파일 통째 **overwrite** → `to_parquet(index=False, engine='pyarrow')`. (`_save_unified_daily` reindex/int 캐스팅 관례 참고.)

10. **public 진입점** (요약 문자열 반환):
    ```python

    def run_order_cross_analysis(date_str, overwrite=True) -> str
    def run_lookback_order_cross_analysis(days=3) -> str

    ```

11. **DAG** (`DB_OrderCrossAnalysis_Dags.py`): `DB_UnifiedSales_Dags.py`의 `resolve_date`/`_conf`/`_truthy`/`_on_failure_callback`(telegram+heal) 패턴 복제. `schedule=DB_ORDER_CROSS_ANALYSIS_TIME`, `catchup=False`, `max_active_runs=1`, `LOOKBACK_DAYS=3`, `default_args`(retries=1, retry_delay=3min). conf: `sale_date`(overwrite), `backfill`, `onedrive_backup`(게이트), 없으면 lookback. 체인: `resolve_date >> build_cross >> backup_onedrive >> validate_cross`.

        import pendulum
    except ModuleNotFoundError:
        from datetime import timedelta; from zoneinfo import ZoneInfo
        now = datetime.now(ZoneInfo("Asia/Seoul"))
        date_iter = ((now - timedelta(days=i)).strftime("%Y-%m-%d") for i in range(1, days + 1))

## Constraints
- **OneDrive 쓰기(`MART_DB` 실제 적재/백업)는 사용자 승인 게이트**: `MART_DB`가 곧 OneDrive이므로 `_save_cross_daily`의 parquet 저장 자체가 OneDrive 쓰기다. Test Case 4~6은 승인 후에만 실행한다.
- 조인키는 반드시 `unified.item_id`(str strip) == 마스터 `상품코드`. 마스터 `수동분류`/`표준_메뉴명`은 NULL이라 사용 금지.
- 멀티메인 과대추정: order에 메인 2개+ 이면 옵션이 각 메인에 중복 귀속됨 → `is_multi_main` 플래그로 표기(제거하지 말 것, 시각화에서 필터). okpos `(order_id, menu_name)` 정밀 grain은 후속 hook만 남기고 1차는 order 단위 통일.
- `total_price < 0`(환불) 행은 co_amount 왜곡 → 집계 전 제외(`sale_type != '취소'` + 음수 방어).
- print 금지, 하드코딩 경로/스케줄 금지, 기존 유틸 재사용 우선.
