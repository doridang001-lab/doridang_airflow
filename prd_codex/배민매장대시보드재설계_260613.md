# 배민 매장 대시보드 재설계 (push 방식 + 검증 + 2패널 반응형)

## 개요
현재 `http://localhost:8787/db-beamin-macro`는 Airflow 로그 정규식 파싱으로 상태를 추측해 표시한다.
- 매장명이 중복 노출될 수 있음
- 수집 순서(`now` → `woori` → `change` → `orders` → `ad`)와 모순되는 상태가 기록될 수 있음
- 변경이력/소요시간 집계가 실제 진행 상태와 어긋날 수 있음

목표는 파이프라인이 단계별 진행을 JSON으로 직접 기록하고( push ),
대시보드는 그 JSON만 읽어 상태를 렌더링하도록 전환해 위 문제를 제거하는 것이다.
동일 맥락에서 화면은 2패널 반응형으로 정리한다.

## Project Conventions
- `logger = logging.getLogger(__name__)` 사용, `print` 금지(일회성 스크립트 제외)
- 임포트는 절대 경로: `from modules.transform.utility.xxx import ...`

- 경로 상수: `from modules.transform.utility.paths import DASHBOARD_DB`

## 구현 가이드

### 1) `beamin_progress.py` (`ProgressReporter`)
#### 저장 위치
`DASHBOARD_DB / "db_beamin_macro_progress.json"`  
쓰기 전략: temp 파일 생성 후 `os.replace`로 atomic write.

#### 스테이지/상태
- `STAGES = ["now", "woori", "change", "orders", "ad"]`
- 상태: `대기`, `수집중`, `완료`, `건너뜀`, `실패`

#### 저장 스키마(예시)
```json
{
  "run_id": "...",
  "started_at": "ISO8601",
  "updated_at": "ISO8601",
  "ended_at": null,
  "stores": {
    "백석점": {
      "store_name": "백석점",
      "seq": 1,
      "brands": {
        "도리당": {
          "account_id": "mjps9142",
          "stages": {
            "now":   {"status": "완료",   "started_at": "...", "ended_at": "...", "note": ""},
            "woori": {"status": "수집중", "started_at": "...", "ended_at": null, "note": ""},
            "change":{"status": "대기",   "started_at": null,  "ended_at": null,   "note": ""},
            "orders":{"status": "대기",   "started_at": null,  "ended_at": null,   "note": ""},
            "ad":    {"status": "대기",   "started_at": null,  "ended_at": null,   "note": ""}
          }
        },
        "나홀로": {
          "account_id": "mjps9142",
          "stages": { "now": ..., "woori": ..., "change": ..., "orders": ..., "ad": ... }
        }
      }
    }
  }
}
```

#### 핵심 메서드 동작
- `__init__(run_id, root=DASHBOARD_DB)`
  - 새 실행마다 기존 문서를 초기화(이전 런 잔재 제거)
  - `started_at`, `updated_at` 즉시 기록
- `seed_store(store, brand, account_id)`
  - 매장 미존재 시 `seq` 증가 순으로 등록
  - 각 브랜드의 5개 스테이지를 `대기`로 초기화(이미 존재하면 덮어쓰기 금지)
- `stage_start(store, brand, stage)`
  - 해당 스테이지를 `수집중`, `started_at` 세팅
  - 현재 스테이지 이전 단계는 `수집중/대기`이면 `완료`로 자동 승격(순차성 보정)
  - 이미 `실패`/`건너뜀`인 단계는 건드리지 않음
- `stage_done(store, brand, stage, status="완료")`
  - `ended_at` 및 상태(`완료`/`건너뜀`) 기록
- `brand_failed(store, brand, note="")`
  - 해당 브랜드에서 `완료`/`건너뜀`가 아닌 단계 모두 `실패` 처리
  - `note` 저장
- `finish()`
  - `ended_at` 설정
  - 남은 `수집중` 상태는 `완료` 처리(작업 중단 없이 마무리 정합성 확보)
- 각 mutation 이후 `updated_at` 갱신 후 즉시 write
- **모든 public 메서드는 best-effort**: 내부 `try/except`로 기록 실패가 수집 파이프라인을 중단시키지 않음

### 2) `DB_Beamin_combined.py` (`collect_now_and_woori`)
- 시그니처:
  - `def collect_now_and_woori(account_list, target_date=None, stability_profile=None, run_id=None):`
- 함수 시작 시 `ProgressReporter` 초기화
- 계정별 `store_list` 확정 직후, store/brand/계정 기준으로 `seed_store` 수행
- 각 수집 섹션 시작 지점에 `stage_start` 호출
  - now 헤더 후: `now`
  - 우리가게 헤더 후: `woori`
  - 변경이력 헤더 후: `change`
  - 주문내역 헤더 후: `orders`
  - 광고 헤더 후: `ad`
- `orders_result.get("ok")`이 False면 `brand_failed(..., "orders failed")`
- 광고 수집 결과 False면 `brand_failed(..., "ad failed")`, True면 `stage_done(..., "ad")`
- 예외/로그인 실패 경로에서 해당 brand에 대해 `brand_failed(..., str(exc))`
- 함수 종료 직전에 `reporter.finish()`

### 3) `DB_Beamin_Macro_Dags.py` (`collect_all`)
- 기존 파이프라인 호출에 `run_id=dag_run.run_id` 추가
- 기존 `write_runtime_metrics` 호출은 유지

### 4) `db_beamin_macro_dashboard.py` (`StoreProgressParser` 제거 + `ProgressReader`)
- 제거 대상
  - `StoreProgressParser`, `_parser_*` 계열 monkeypatch
  - `SECTION_START_RE` 등 로그 정규식 기반 상태 파싱 로직
  - `AirflowMetadataRepository`의 로그 의존 부분(런 요약은 기존 메타 유지)
- 신규 `ProgressReader`
  - `load()`: `DASHBOARD_DB/db_beamin_macro_progress.json` 로드 (없으면 빈 문서)
  - `_agg_stage(brand_states, stage)` 집계 규칙  
    - 실패가 하나라도 있으면 `실패`
    - 수집중이 하나라도 있으면 `수집중`
    - 대기가 하나라도 있으면 `대기`
    - 모두 `완료/건너뜀`이면 `완료` (전부 `건너뜀`이면 `건너뜀`)
  - 매장 행 생성
    - `store_key = store_name`
    - `brand = "/".join(present_brands)`
    - 각 컬럼은 `_agg_stage` 적용, `brand_states`(브랜드별 상세 배지 데이터) 포함
  - `store_state` 계산
    - 모든 브랜드 5단계가 `완료/건너뜀`이면 `완료`
    - 하나라도 실패면 `실패`
    - 하나라도 수집중이면 `수집중`
    - 그 외는 `부분완료` 또는 `대기`로 표기
  - `elapsed_sec`: 전체 브랜드의 `min(started_at) -> max(ended_at or now)` (ISO 파싱)
  - 정렬: `seq` 오름차순
  - 실패 로그 수집:
    - `status == "실패"`인 항목을
      `{"time","store_name","brand","stage","message","level":"WARN"}` 형식으로 누적
- `SnapshotBuilder.build`
  - stores/live logs는 `ProgressReader` 결과 사용
  - 런 요약(active/평균/예상 완료)은 기존 Airflow 메타 우선, 실패 시 progress의 `started_at/ended_at` 폴백
- `SnapshotStore`, SSE, HTTP 라우트/서버 루틴은 유지

## Reference

### `beamin_stability.py` 패턴
```python
import json
from datetime import UTC, datetime
from pathlib import Path
from modules.transform.utility.paths import DASHBOARD_DB

def write_runtime_metrics(*, run_id, stage, payload, root=None):
    target_root = Path(root or DASHBOARD_DB)
    target_root.mkdir(parents=True, exist_ok=True)
    path = target_root / f"db_beamin_macro_{stage}_metrics.json"
    document = {"generated_at": datetime.now(UTC).isoformat(), "run_id": run_id, "stage": stage, **payload}
    path.write_text(json.dumps(document, ensure_ascii=False, indent=2), encoding="utf-8")
    return path
```

## Verification Loop
1. 테스트 1~6 실행
2. 실패 건:
   - 2순위: 순차성/집계 관련 실패 우선 점검
   - 3순위: 삭제 잔재로 인한 `ImportError` 점검

4. PASS 시 종료

## 제약 조건
- 기록은 best-effort. 기록 실패가 수집 자체를 중단하면 안 된다.
- 경로는 반드시 `DASHBOARD_DB` 상수 기반
- `StoreProgressParser` 및 로그 정규식 의존은 완전 제거
- `store_key`는 `store_name` 그대로 사용 (브랜드 접두어 없음)
- 하드코딩된 브랜드 개수 금지(브랜드 수는 실제 데이터 기준)
- `started_at`/`ended_at`은 ISO 기반; 시작 시각을 wall-clock 재계산하지 않음
- 기존 `db_beamin_macro_*_metrics.json`, snapshot 호환 유지
