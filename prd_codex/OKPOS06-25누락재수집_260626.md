# OKPOS 2026-06-25 누락 재수집(보완)

## 이슈 가정 (체크 후 확정)
- `DB_OKPOS_Sales_Dags.py` 정기 실행(07:15 UTC 기준 07:15 KST)은 동작했지만,
  06-25 데이터는 `_NO_DATA` 마킹으로만 처리되어 적재되지 않음.
- 로그 근거: save_to_raw에서 06-25 4개 매장의 `today/receipt/daily`가 각각
  `매출 없음 → NO_DATA 마킹`으로 종료됨.
- `check_and_fill_missing_today`는 같은 로직으로 06-25 4개 조합을 재다운로드했지만,
  이번에는 `재다운로드 결과 매출 없음`으로 다시 `_no_data__*.txt`가 갱신됨.
- 결론: 해당 시점 기준으로는 **오차(비정상)라기보다 OKPOS 미마감/헤더-빈파일 패턴으로 인한 NO_DATA 누락 패턴**에 가깝고, 즉시 성공 적재는 기대되지 않음.

## 코드 보완 결과 요약
- `_transform_okpos_daily_df`가 `영업일수합: 0 일`을 감지하면 daily 쪽만 빈 DataFrame으로 간주하고 `daily`에서만 NO_DATA 처리한다.
- `save_to_raw`는 `today/receipt/daily`를 **페이지별로 독립 처리**하므로, `daily=0` 자체가 `today`/`receipt` 적재를 직접 차단하지 않는다.
- 06-25 현상은 `daily=0`이 직접적인 차단인 것보다, 3개 페이지가 모두 NO_DATA 판정되어 저장 건수가 0으로 끝난 패턴이 맞다.

## 성공 기준(리캡)
- 06-25 네 매장 `okpos_daily.csv`, `okpos_order.csv`, `okpos_order_item.csv` 각각 `sale_date='2026-06-25'` 행이 1개 이상 존재
- `daily/order/receipt`의 행 수/금액 정합성(가능 범위 내)을 통과
- `.no_data__okpos_*.txt`에서 06-25가 제거
- (필요 시) `DB_UnifiedSales` 수동 트리거 성공

```bash
            row[f] = "NOFILE"

    # 파일은 store 단위이므로 grep으로 한 번에 점검
    for f in p:
        pass
PY
```

### 2) 동일 날짜 단일 강제 재수집(권장 1차)
```bash
docker compose exec airflow-scheduler airflow dags trigger DB_OKPOS_Sales_Dags \
  --conf '{"sale_date_from":"2026-06-25","sale_date_to":"2026-06-25","force_redownload":true,"replace_by_date":true,"trigger_unified_sales":true}'
```

- `force_redownload=true` : 오늘/영수증/일자별을 재다운로드
- `replace_by_date=true` : 기존 06-25 데이터가 있으면 해당 날짜만 덮어쓰기
- `trigger_unified_sales=true` : 수집 성공 시 `DB_UnifiedSales` 동시 갱신

## 점검 항목(필수)
```bash
python -X utf8 - <<'PY'
from pathlib import Path
from modules.transform.utility.paths import RAW_OKPOS_SALES
```

## 트러블슈팅 기준
- `save_to_raw` 로그에서 `save_to_raw 완료: 0개`가 반복되면 **정상 패턴인지(헤더/영업일수합 0)** 먼저 확인.
- 3회 연속으로 `영업일수합: 0 일` 또는 빈 헤더만 내려오면, OKPOS 측 수집 상태 문제로 판단.
- 장기 미수집이 반복되면:
  - 크론 타이밍(7:15) 이외의 시간대에서 1회 수동 재수집
  - 브라우저 로그인/권한 세션 상태 로그인 실패 여부 점검
  - 필요 시 `DB_OKPOS_Sales_Dags.py`에서 `_NO_DATA_RETRY_RECENT_DAYS` 및 로그인/다운로드 로그 패턴 보강 검토
