### B. 10057 차단 매장 → 재수집 제외

**B-1. `03_coupangeats.js` 매장별 throttle 발생 추적**
`_isOrdersThrottleVisible()`(L83)는 이미 `10056/10057/ACCESS_DENIED/단시간 내 반복 요청/과도한 요청`을 감지함. 매장 단위로 "이 매장에서 차단이 발생했는가"를 판정해야 함.

배치 멀티매장 루프(`_collectOrdersMultiStore`, L2195~2297) 안에서 매장 처리 시작 시 throttle 카운트를 스냅샷하고, 처리 종료 시 비교:
```js
// for 루프 안, storeName 확정 직후
const throttleCountBefore = this._ordersThrottleCount;
const blockedNow = () =>
  this._isOrdersThrottleVisible() || this._ordersThrottleCount > throttleCountBefore;
```

**B-2. `03_coupangeats.js` 차단 시 storeResults에 `blocked` 플래그 + 즉시 다음 매장**
배치 루프에서 `collected` 실패 분기(L2284) 및 데이터 로드 직후, 차단이면 55~210초 cooldown을 타지 말고 즉시 `blocked` 처리:
```js
// collected 판정 직후
if (!collected) {
  if (blockedNow()) {
    Utils.updateProgressModal({ debug: '⛔ 쿠팡 차단(10057) 감지 — 재수집 제외, 다음 매장' });
    Utils.updateMultiStoreStoreResult('error', '⛔ 쿠팡 차단(10057) — 재수집 제외');
    storeResults.push({ storeName, completed: false, status: 'skip', blocked: true,
      note: '쿠팡 차단(10057) — 재수집 제외' });
    continue;
  }
  Utils.updateMultiStoreStoreResult('error', '수집 결과 없음');
  storeResults.push({ storeName, completed: false, status: this._stopFlag ? 'warn' : 'fail', error: true, note: this._stopFlag ? '중단됨' : '수집 결과 없음' });
  continue;
}
```
단일매장 경로(L2092, L2164 `storeResults: [{...}]`)에도 동일하게 `blocked` 분기를 추가: collected 실패 + `_isOrdersThrottleVisible()`이면 `blocked: true, status: 'skip'`로 push.

> 차단 매장은 `_waitBeforeOrdersSearch`의 cooldown 대기(55~210초)를 더 이상 반복하지 않고 즉시 건너뛴다. (차단 판정 후 `continue`로 다음 매장 진행)

**B-3. `runner.js` `recordCategoryResults`에 차단 기록 (L185~202)**
`result.blocked`면 row에 `${category}Blocked: true` 기록 (`ordersOk`는 false 유지):
```js

row[`${category}Ok`] = completed;
if (result && result.blocked) row[`${category}Blocked`] = true;

```
`resetStoreStatus`(L141) / `ensureStoreStatus`(L159) 초기 객체에 `ordersBlocked:false, cmgBlocked:false, menuBlocked:false` 필드 추가(선택, 안전성).

**B-4. `runner.js` `incompleteStores`에서 차단 매장 제외 (L215)**
재시도 루프와 백필 큐(L1204 `ce_backfill_queue`)가 모두 이 함수를 쓰므로 한 곳만 고치면 양쪽 제외됨:
```js
function incompleteStores() {
  return [...storeStatusRows.values()]
    .filter(r => {
      const ordersDone = r.ordersOk || r.ordersBlocked;
      const cmgDone    = r.cmgOk    || r.cmgBlocked;
      const menuDone   = r.menuOk   || r.menuBlocked;
      return !(ordersDone && cmgDone && menuDone);
    })
    .map(r => r.store);
}
```
> 의미: orders가 차단되어도 cmg/menu가 남았으면 그 매장은 여전히 재시도 대상이지만, **차단된 orders 카테고리 자체는 재시도하지 않는다**(skip 플래그가 `relatedRows.every(r => r.ordersOk)` 기준 — 차단 매장의 ordersOk는 false이므로 orders skip이 안 걸릴 수 있음. 따라서 retry의 skip 계산도 blocked를 done으로 간주하도록 L1136 보정 권장).

**B-5. `runner.js` 재시도 skip 계산 보정 (L1135-1139)**
```js
const skip = {
  orders: relatedRows.every(r => r.ordersOk || r.ordersBlocked),
  cmg:    relatedRows.every(r => r.cmgOk    || r.cmgBlocked),
  menu:   relatedRows.every(r => r.menuOk   || r.menuBlocked)

```

**B-6. (선택) 로그/배지** — 차단 매장은 `setRow(i, {orders:'skip', note:'쿠팡 차단(10057)'})`로 표기하고 최종 요약 로그에 "차단 N개" 구분. 기능에 필수는 아님.

## Constraints
- **두 위치(개발용 + build 사본) 항상 동일 변경**. 한쪽만 고치면 안 됨.
- 차단 매장은 cooldown(55~210초) 재시도를 타지 않고 **즉시 다음 매장으로** 진행해야 함 (시간 낭비 방지가 핵심 목적).
- `ordersOk`는 차단 시에도 false 유지 (실제 수집 안 됨). 제외는 `ordersBlocked` 별도 플래그로만 처리.
- 기존 throttle 감지 함수 `_isOrdersThrottleVisible()` 재사용 — 새 감지 로직 만들지 말 것.
- manual 단일/배치 멀티 양 경로 모두 blocked 처리 반영.
- 페이스 강도는 "중간"(계정 6~12초, 조회 22~35초, manual 1.2배) — 과하게 늘리지 말 것.
