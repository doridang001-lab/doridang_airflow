# 쿠팡 지정날짜 수집 버튼 추가

## Task
쿠팡이츠 확장 프로그램(`c:\Users\민준\OneDrive - 주식회사 도리당\Extention\doridang_collector_개발용`)의 `runner.html`("쿠팡 전체 자동수집" 대시보드)에는 오늘/어제 기준 전체 배치 버튼만 있다. 매장 1곳을 선택해 임의의 시작일~종료일(예: 2026-01-01~2026-01-03) 범위로 **주문서(orders)만** 재수집하는 새 버튼("지정날짜 수집")을 추가한다. CMG(광고)·메뉴 수집은 절대 실행되지 않아야 한다.

**중요 배경 (버그 회피 이유)**: 기존 오늘/어제 버튼이 쓰는 배치 경로(`source:'batch'`)는 전역 체크포인트(`chrome.storage.local['ce_orders_checkpoint']`, 날짜 하나로만 키가 잡힘)와 다운로드파일 beacon 기반 중복수집 방지를 쓴다. 지정날짜 수집의 종료일이 정규 배치가 이미 처리한 날짜와 겹치면 "이미 완료됨"으로 오판해 아무것도 수집하지 않고 스킵되는 치명적 버그가 생긴다(재수집이 목적인 기능인데 재수집이 막힘). 반대로 지정날짜 수집이 체크포인트를 저장하면 다음날 정규 배치가 같은 날짜를 스킵하는 역방향 오염도 생긴다.
→ 따라서 배치 경로를 재사용하지 말고, 체크포인트/beacon을 전혀 쓰지 않는 **manual 경로**(`opts.source !== 'batch'`)를 확장한다. manual 경로에는 이미 `_setOrdersDateRange(startDateObj, endDateObj)`(진짜 범위를 페이지 DayPicker에 설정, 에러 이어수집 로직에서 이미 검증됨)와 `_selectStoreInOrdersDropdown(storeText)`(이미 현재 매장과 같으면 즉시 true 반환, 다르면 드롭다운 전환)가 존재하므로 이 둘을 조합만 하면 된다.

async function runCustomDateOrders(storeName, startStr, endStr) {
  if (running) { log('다른 배치 실행 중 — 지정날짜 수집 불가', 'err'); return; }
  if (!storeName) { log('지정날짜 수집: 매장을 선택하세요', 'err'); return; }
  if (!startStr || !endStr) { log('지정날짜 수집: 시작일/종료일을 입력하세요', 'err'); return; }
  if (startStr > endStr) { log('지정날짜 수집: 시작일이 종료일보다 늦습니다', 'err'); return; }

  running = true; stopRequested = false;
  resetStoreStatus([storeName]);
  el.startBtn.disabled = true; el.todayBtn.disabled = true; el.topHalfBtn.disabled = true;
  el.bottomHalfBtn.disabled = true; el.topHalfTodayBtn.disabled = true; el.bottomHalfTodayBtn.disabled = true;
  el.stopBtn.disabled = false;
  startTs = Date.now();
  elapsedTimer = setInterval(() => { el.mElapsed.textContent = fmtMs(Date.now()-startTs); }, 1000);

  log(`지정날짜 수집 시작: ${storeName} (${startStr} ~ ${endStr}) — 주문서만`, 'info');

  let workTab;
  try {
    workTab = await chrome.tabs.create({ url: 'about:blank', active: true });
    const runIndices = targetRunIndices([storeName]);
    for (const qi of runIndices) {
      if (stopRequested) break;
      let res;
      try {
        res = await processAccount(qi, workTab.id, {}, [storeName], {
          customOrdersRange: { store: storeName, start: startStr, end: endStr }
        });
      } catch(e) {
        log(`[${queue[qi].id}] 지정날짜 수집 예외: ${e.message}`, 'err');
        res = { status:'fail', note:e.message };
      }
      updateMeta();
      if (!stopRequested) await waitBeforeNextAccount(res);
    }
  } catch(e) {
    log(`지정날짜 수집 예외: ${e.message}`, 'err');
  } finally {
    if (workTab) { try { await chrome.tabs.remove(workTab.id); } catch(_) {} }
    finishBatch();
  }
}

### runner.js — `allStoreNamesSorted`, `targetRunIndices` (기존, 그대로 재사용)
```js
function allStoreNamesSorted() {
  const names = [];
  queue.forEach(acc => acc.stores.forEach(store => names.push(store)));
  return uniq(names).sort((a, b) => a.localeCompare(b, 'ko'));
}

function accountTargetStores(acc, targetStores = runTargetStores) {
  const allowedStores = uniq(acc.stores);
  const targetSet = new Set((targetStores || []).map(storeCompareKey).filter(Boolean));
  if (!targetSet.size) return allowedStores;
  return allowedStores.filter(store => {
    const key = storeCompareKey(store);
    return [...targetSet].some(target => key === target || key.includes(target) || target.includes(key));
  });
}

function targetRunIndices(targetStores = runTargetStores) {
  const picked = [];
  queue.forEach((acc, i) => {
    if (accountTargetStores(acc, targetStores).length > 0) picked.push(i);
  });
  return picked;
}
```

## Verification Loop

```
LOOP until all PASS:

  2. FAIL 항목 → 원인 분석
  3. 코드 수정
  4. 전체 Test Cases 재실행
종료 조건: 전체 PASS + Constraints 위반 없음
```

## Constraints
- 기존 오늘/어제/상위50%/하위50% 버튼이 쓰는 `_setOrdersDateToYesterday`, `_resolveBatchTargetDate`, 체크포인트(`ce_orders_checkpoint`)/beacon 로직, `processAccount`의 기존 스텝 흐름은 **한 줄도 수정하지 않는다**.
- `content/03_coupangeats.js:3094`의 `_setOrdersDateRange`, `content/03_coupangeats.js:2844`의 `_selectStoreInOrdersDropdown`은 이미 검증된 기존 함수이므로 **재구현하지 말고 그대로 호출**한다.
- `customOrdersRange` 분기 안에서 `_loadOrdersCheckpoint`/`_saveOrdersCheckpoint`/`checkDownloadBeacon`을 절대 호출하지 않는다 (체크포인트 충돌 버그 재도입 방지가 이 기능의 핵심 요구사항).
- CMG/메뉴 수집은 `processAccount`의 early return으로 코드 경로 자체를 차단해야 하며, 단순히 `skip:{cmg:true, menu:true}` 플래그에만 의존하지 않는다.
- `runner.js`의 새 버튼(`#customDateBtn`) 및 관련 로직은 `#startBtn`/`#todayBtn` 등 기존 버튼의 DOM id를 변경하거나 제거하면 안 된다 (외부 자동화 스크립트 `coupang_runner_autoclick.py`가 기존 id로 버튼을 클릭하는 것으로 문서화되어 있음 — `C:\airflow\docs\coupang-boot-automation.md` 참고).
