# 배민 매크로 수집 근본안정화 개선안

## Task
`DB_Beamin_Macro_Dags`의 `collect_all`은 크롬/셀레늄 세션 사망으로 1패스 계정 완결률이 구조적으로 1 미만이라(최근 scheduled 8회 중 100% 성공 1회뿐), 임시 뒷수습 장치인 `DB_Beamin_Macro_Dags_Retry`가 상시 필수 경로가 되어 있다. 1패스 완결률을 1에 근접시켜 Retry를 "예외적 안전망"으로 되돌리는 것이 목표다. 결정적 반례: 이미 강한 `safe_memory` 프로필로 돈 07-10이 오히려 최악(런타임 5.3h·크래시 162)이었으므로 "강한 프로필 전환"만으로는 역효과다. 해결은 (A) 로그인/세션 유실의 in-run 즉시 복구 + (B) 1회 실행 길이 축소를 함께 잡는 것.

## Implementation Steps

### Phase 1 — 로그인/세션 유실의 in-run 즉시 재시도 (최고 레버리지·저위험)
1. `DB_Beamin_combined.py`의 `collect_now_and_woori`(235~) 메인 루프에서, 계정 유실 사유를 **login-stage**(로그인 2회 실패 / 대시보드 매장목록 로드 실패 / 매장 목록 조회 예외)와 **collection-stage**(orders·ads 등 중간 유실)로 구분해 라벨링한다.
   - login-stage 유실분은 `failed_accounts`에 담되, 별도 리스트(예: `login_lost_accounts`)로도 수집.
2. 메인 루프 종료 직후, `login_lost_accounts`만 대상으로 **second pass 1회** 순회를 추가한다.
   - 재사용 유틸: `_build_dashboard_session`(combined.py:176), `_ensure_dashboard_store_select`(combined.py:206) — 이미 세션 재생성·매장목록 대기 복구 로직을 포함.
   - second pass 진입 전 계정당 랜덤 대기(예: 60~120초) — RemoteDisconnected는 수 분 뒤 재시도 시 대부분 회복.
   - second pass에서 성공하면 `failed_accounts`에서 제거하고 `success`로 반영, 여전히 실패면 그대로 둔다.
3. orders/ads 중간 유실은 기존 `retry_once_failed` 경로를 그대로 유지한다(second pass는 login-stage 한정).

### Phase 2 — 적정 복원력 프로필 `safe_daily` 신설
4. `beamin_stability.py`의 `STABILITY_PROFILES`에 프로필 추가:
   ```python
   "safe_daily": {

       "orders_validation_retry": 2,
       "max_parallel_accounts": 1,
       "driver_restart_every_stores": 3,        # safe_memory의 1은 과도(런타임 폭증) → 3으로 절충
       "login_attempts": 3,                     # safe의 2 → 3 (로그인 회복력↑)
       "login_retry_wait_range": (5.0, 15.0),
       "max_session_recovery_per_account": 3,   # safe의 2 → 3
   },
   ```
5. `DB_Beamin_Macro_Dags.py`의 `collect_all`(723~)에서 scheduled 실행이 프로필 미지정으로 `safe`를 쓰는 현 동작을 `safe_daily` 기본 적용으로 바꾼다.
   - 최소 변경안: `resolve_stability_profile(conf.get("stability_profile") or "safe_daily")` 형태로 기본값 주입. `retry_failed`(319~) 호출부도 동일 기본값으로 맞춘다.

### Phase 3 — 배치 분할로 1회 실행 길이 상한 (구조적)
6. `schedule.py`에 배치 staggered 스케줄 상수 추가(예: 03:15 / 04:30 / 05:45). 기존 `SMD_BAEMIN_COLLECT_TIME`("15 3 * * *")은 배치1로 유지하거나 배치 상수로 확장.
7. `DB_Beamin_Macro_Dags.py`에서 기존 `_split_accounts_by_range`(132) / `COLLECT_RANGE` / `conf["collect_range"]` 경로를 재사용해 스케줄별로 다른 배치를 수집하도록 배선한다. 단일 DAG + 스케줄별 `collect_range` conf 주입이 최소 변경.
8. **필수 동반 수정**: 배치화 시 `validate_toorder`(851~)가 아직 안 돌린 배치의 매장을 배민=0으로 오판하지 않도록, 검증 대상을 해당 배치 소속 매장으로 한정하거나 전체 배치 완료 후 1회만 검증하도록 스코프를 좁힌다.

## Verification Loop
구현 완료 후 아래 루프를 **모든 Test Cases PASS까지** 반복한다.

```
LOOP until all PASS:
  1. Test Cases 순서대로 실행
  2. FAIL 항목 → 원인 분석
  3. 코드 수정
  4. 전체 Test Cases 재실행
종료 조건: 전체 PASS + Constraints 위반 없음
```

## Constraints
- Phase 3(배치화)은 `validate_toorder` 검증 스코프 한정과 **반드시 함께** 적용한다. 빠뜨리면 오탐(불일치) 폭증.
- `DEFAULT_PROFILE`("safe")은 바꾸지 말 것 — 다른 호출부 호환. scheduled 기본은 DAG 호출부에서 `safe_daily`를 명시 주입한다.
- second pass는 login-stage 유실 한정. orders/ads 중간 유실은 기존 `retry_once_failed`/Retry DAG 경로 유지(이중 재시도 방지).
- 크롬 옵션(`croling_beamin.py`)은 이미 OOM 대책 풀장착 상태 → 건드리지 말 것.
- 폴더 구조/파일명 변경 금지, 기존 import 경로 호환 유지.
- `safe_daily`의 `driver_restart_every_stores`는 3에서 시작해 실측(런타임 vs 크래시)으로 튜닝.
