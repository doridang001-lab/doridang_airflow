## 근거 (진단 결과 — 재조사 불필요)

수집 로그(`raw row html sample`)가 잡아낸 실제 DOM:

```html
<tr class="fc-list-item js-mouseover atd-attending" mouseover-text="[요청] [전략기획부] 매출 통합 작업">
  <td class="fc-list-item-time">종일</td>
  <td class="fc-list-item-marker"><span class="fc-event-dot"></span></td>
  <td class="fc-list-item-title"><a>[요청] [전략기획부] 매출 통합 작업</a></td>
</tr>
```

사용자가 같은 화면·같은 셀렉터에서 보는 DOM:

```html
<td class="fc-list-item-title"><a>[완료] [전략기획부] 매출 통합 작업</a></td>
<td class="fc-list-item-title"><a>[진행] Claude 개인 AI 비서 기획서</a></td>
```

같은 listMonth 뷰인데 접두 태그만 다르다 → **뷰가 아니라 필터 상태 차이**다.

수집 로그: `Flow Calendar 필터 선택: 일정전체=True 업무전체=True` — 두 버튼 모두 무조건 클릭했다. 이 버튼들은 토글이므로 이미 켜져 있던 "업무 전체"가 꺼진다.

`[요청]`은 `_CALENDAR_NON_STATUS_PREFIX_TAGS`에 의해 건너뛰어지고, 그 뒤에는 `[전략기획부]`·`[매크로]` 같은 부서 태그만 남아 상태 매칭이 실패 → 전부 기본값 `"진행"`으로 떨어진다.

전체 브리핑 로그를 통틀어 `[진행]`·`[피드백]`·`[결제중]` 태그는 한 번도 수집된 적이 없다. **별칭 테이블만 고쳐서는 해결되지 않는다.**

순수 함수 검증: 실제 제목 12건을 `_dedupe_briefing_calendar_events` → `_format_calendar_sections`에 직접 통과시키면 진행 3 / 피드백 1 / 결제중 1 / 테스트 1로 정확히 나뉘고 `[완료]`도 걸러진다. **분류 로직 자체는 이미 정상이다.**

## Implementation Steps

### 1. 필터 선택을 토글이 아니라 "원하는 상태로 보장"으로 (근본 원인)

`_select_flow_calendar_filters` (pipeline:195). 현재 JS가 `clickEl()`로 무조건 클릭한다.

활성 표시 class를 먼저 읽고 **이미 활성이면 클릭하지 않는다**. 활성 class명은 실행 환경에서 확정해야 하므로(`on` / `active` / `selected` 후보) JS에서 `className`을 그대로 반환해 로그로 남기고, 후보 집합에 대해 판정한다.

```js
function isActive(el) {
  var c = ' ' + (el.className || '') + ' ';
  return c.indexOf(' on ') >= 0 || c.indexOf(' active ') >= 0 || c.indexOf(' selected ') >= 0;
}
function ensureActive(el) {
  if (!el) return {found: false};
  var before = el.className || '';
  if (!isActive(el)) {
    try { el.click(); }
    catch (e) { el.dispatchEvent(new MouseEvent('click', {bubbles:true, cancelable:true, view:window})); }
  }
  return {found: true, before: before, after: el.className || ''};
}
var scheduleAll = document.querySelector('#scheduleFilter .js-filter-button[gubun="0,1"]');
var taskAll = document.querySelector('#taskFilter .js-filter-button[gubun="2"]');
return {schedule: ensureActive(scheduleAll), task: ensureActive(taskAll)};
```

Python 쪽에서 반환된 `before`/`after` className을 `logger.info`로 남긴다 — 활성 class명 확정에 필요하다.

### 2. 수집 결과 검증 + 자동 복구

`_collect_calendar_events_inner` (pipeline:100). listMonth 추출 직후, 수집된 제목 중 상태 태그(`_CALENDAR_STATUS_ALIASES` 또는 `_CALENDAR_EXCLUDED_STATUS_TAGS`의 키)를 가진 것이 **하나도 없으면** 필터를 반대로 토글하고 한 번 더 추출한다. 두 번째도 없으면 `logger.warning`으로 명시적 경고를 남기고 그때 수집한 결과를 쓴다.

지금은 태그가 없어도 조용히 전부 `진행`이 되어 문제가 몇 주간 드러나지 않았다.

```python
def _events_have_status_tags(events: list[dict]) -> bool:
    known = set(_CALENDAR_STATUS_ALIASES) | _CALENDAR_EXCLUDED_STATUS_TAGS
    for event in events or []:
        for tag in re.findall(r"\[([^\]]+)\]", event.get("summary") or ""):
            if tag.strip() in known:
                return True
    return False
```

주의: 이 검사는 `_dedupe_briefing_calendar_events`가 접두어를 제거하기 **전**의 raw 이벤트에 적용해야 한다. `_extract_flow_calendar_events_from_html`이 이미 dedupe를 호출해 반환하므로, raw 리스트를 따로 얻을 수 있게 분리하거나 dedupe 결과의 `status_source`가 전부 `"default"`인지로 판정한다 (후자가 더 간단하다).

### 3. 무태그 이벤트를 조용히 삼키지 않기

`_extract_calendar_event_status` (pipeline:372)가 무조건 `"진행"`을 반환한다. 기본값 자체는 유지하되, 태그를 못 찾은 제목 목록을 `logger.warning`으로 남긴다.

### 5. 상태를 두 번 유도하지 않기 (재발 방지)

`_format_calendar_sections` (pipeline:425)가 이미 접두어가 제거된 `summary`에서 태그를 다시 파싱하려 시도하고, 실패하면 `status_source == "tag"` 폴백에 기댄다. 이 이중 유도가 같은 계열 버그를 숨긴다.

`_dedupe_briefing_calendar_events`가 이미 확정한 `event["status"]`를 그대로 신뢰하도록 단순화한다.

```python
def _format_calendar_sections(events: list[dict]) -> str:
    groups = {label: [] for label in _CALENDAR_SECTION_LABELS}
    for event in events or []:
        status = _normalize_calendar_event_text(event.get("status")) or "진행"
        groups.setdefault(status if status in groups else "진행", []).append(event)

    blocks = []
    for label in _CALENDAR_SECTION_LABELS:
        items = groups.get(label) or []
        body = "\n".join(f"  {e['time']}  {e['summary']}" for e in items) if items else "  (없음)"
        blocks.append(f"## {label}\n{body}")
    return "\n".join(blocks)
```

### 6. weekView fallback JS 정합성

`_collect_flow_calendar_events_from_week_view` 내부 JS `statusOf` (pipeline:539)가 Python 별칭 테이블과 따로 논다 — `[요청]`을 피드백으로 매핑하고 `세부업무`를 진행으로 접는다.

JS는 원본 태그 문자열만 그대로 실어 보내고, 매핑은 Python 별칭 테이블 한 곳에서만 하도록 한다.

```js
function statusOf(title) {
  var match = (title || '').match(/^\s*\[([^\]]+)\]/);
  return match ? match[1].trim() : '';
}
```

이 fallback은 로그상 항상 0건이라 실질 동작은 안 하지만, 살아나면 잘못 분류한다.

## Verification Loop

구현 완료 후 아래 루프를 **모든 Test Cases PASS까지** 반복한다.

```
LOOP until all PASS:
  1. Test Cases 1→5 순서대로 실행
  2. FAIL 항목 → 원인 분석
  3. 코드 수정
  4. 전체 Test Cases 재실행
종료 조건: 전체 PASS + Constraints 위반 없음
```

Test Case 4가 FAIL이면(여전히 `[요청]` 수집) 활성 class 후보(`on`/`active`/`selected`)가 실제와 다른 것이다. 로그에 찍힌 `before` className을 읽어 판정 조건을 고친다 — 추측하지 말고 로그를 근거로 삼는다.

- 파일이 이미 존재하면: 덮어쓰기
- import가 모호하면: Reference Code의 패턴 따라가기
- 타입 힌트 여부: 기존 파일과 동일하게 (`str | None`, `list[dict]` 스타일)
- 주석 여부: 최소화 (WHY만, WHAT 설명 금지)
- 변수명·함수명 스타일: snake_case, private helper는 `_` 접두, 기존 파일과 동일하게
- 활성 class 판정이 애매하면: `on`/`active`/`selected` 세 후보를 모두 검사하고 className을 로그로 남긴다
