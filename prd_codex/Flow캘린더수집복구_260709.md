## Implementation Steps

### 1. page_load_timeout을 로그인 이후로 이동 + 값 완화 — 근본 원인

`_collect_calendar_events_inner()` (약 90행) 의 현재 순서가 문제다:

```python
driver = _launch_flow_calendar_browser()
_configure_flow_driver_timeouts(driver)   # ← 45초 캡을 로그인 전에 건다
driver.set_window_size(1920, 1080)
if not do_login(driver):                  # ← 92~103초 필요 → 50초에 잘림
    logger.warning("Flow 로그인 실패 — 일정 수집 건너뜀")
    return []
```

`_configure_flow_driver_timeouts(driver)` 호출을 `do_login()` **성공 이후**로 옮긴다:

```python
driver = _launch_flow_calendar_browser()
driver.set_window_size(1920, 1080)
if not do_login(driver):
    logger.warning("Flow 로그인 실패 — 일정 수집 건너뜀")
    return []
_configure_flow_driver_timeouts(driver)   # 로그인 성공 후에 건다
```

로그인 자체는 `do_login()` 내부의 `FLOW_LOGIN_SUCCESS_TIMEOUT`(기본 150초) WebDriverWait이 이미 지키므로 별도 캡이 필요 없다.

동시에 `_FLOW_CALENDAR_NAV_TIMEOUT_SEC` (약 37행) 을 **45 → 120초**로 올리고 env 튜닝 가능하게 한다:

```python
_FLOW_CALENDAR_NAV_TIMEOUT_SEC = int(os.getenv("FLOW_CALENDAR_NAV_TIMEOUT_SEC", "120"))
```

> 45초를 유지하면 안 된다. 로그인 후의 `driver.get(calendar_url)` 도 같은 무거운 SPA라서, 두 캘린더 URL이 모두 per-URL `except` 에 걸려 `continue` 되고 결국 `[]` 가 된다.

### 2. 프로세스 캡을 내부 deadline에 맞추기 — 잠재 폭탄

35~36행이 서로 모순이다. 내부 deadline은 600초인데 부모가 120초에 죽인다 (→ 600초 deadline 로직은 도달 불가능한 dead code):

```python
_FLOW_CALENDAR_COLLECTION_TIMEOUT_SEC = 600
_FLOW_CALENDAR_PROCESS_TIMEOUT_SEC = int(os.getenv("FLOW_CALENDAR_PROCESS_TIMEOUT_SEC", "120"))
```

프로세스 캡을 내부 deadline에서 **파생**시켜 다시 어긋나지 않게 한다 (grace 60초):

```python
_FLOW_CALENDAR_COLLECTION_TIMEOUT_SEC = 600
_FLOW_CALENDAR_PROCESS_TIMEOUT_SEC = int(
    os.getenv("FLOW_CALENDAR_PROCESS_TIMEOUT_SEC", str(_FLOW_CALENDAR_COLLECTION_TIMEOUT_SEC + 60))
)
```

660초 < `collect_briefing_data` 의 `execution_timeout=15분`(900초) 이고, 실측 최장 수집이 467초라 여유가 충분하다. **DAG 파일은 수정 불필요.**

### 3. 큐 수신 레이스 제거

`_collect_calendar_events()` 의 `result_queue.get_nowait()` (약 76행) 는 `process.join()` 직후 호출된다. 자식이 종료해도 부모 큐의 리더 스레드가 아직 항목을 올리지 못해 `queue.Empty` 가 튀는 알려진 패턴이다 (그러면 조용히 `[]`).

`get_nowait()` → `get(timeout=10)` 으로 바꾼다. `except queue.Empty` 처리는 그대로 둔다.

### 4. 수집 실패를 브리핑에 명시

지금은 "일정 0건"과 "수집 실패"가 둘 다 섹션 생략으로 끝나 구분이 안 된다.

**4-1.** `_collect_calendar_events()` 가 이벤트 목록과 성공 여부를 함께 돌려주도록 시그니처를 바꾼다 → `tuple[list[dict], bool]`.
실패 경로 4개는 모두 `ok=False` 로 반환:
- 프로세스 타임아웃 (`process.is_alive()`)
- 비정상 종료 (`exitcode not in (0, None)`)
- `payload["ok"] is False`
- `queue.Empty`

정상 경로만 `(events, True)`.

> 주의: `_collect_calendar_events_inner()` 내부의 `return []` (로그인 실패 / deadline 초과) 는 워커에서 `{"ok": True, "events": []}` 로 감싸진다. 로그인 실패를 `ok=False` 로 잡으려면 `_collect_calendar_events_inner()` 가 실패 시 예외를 던지거나, 워커가 별도 플래그를 싣도록 해야 한다. **가장 단순한 방법**: `_collect_calendar_events_inner()` 의 로그인 실패 지점에서 `raise RuntimeError("Flow 로그인 실패")` 를 던지고, 워커의 `except BaseException` 이 `{"ok": False, ...}` 로 잡게 한다. (이번 장애가 정확히 이 경로다.)

**4-2.** `collect_briefing_data()` (약 943행) payload에 `"calendar_ok": ok` 추가:

```python
calendar, calendar_ok = _collect_calendar_events()
...
payload = {
    "calendar": calendar,
    "calendar_ok": calendar_ok,
    "failures": failures,
    ...
}
```

**4-3.** `generate_briefing()` 의 섹션 렌더링 (약 1056~1058행) 을 분기:

```python
if not data.get("calendar_ok", True):
    sections.append("📅 오늘 일정\n  (수집 실패 — collect_briefing_data 로그 확인)")
elif data["calendar"]:
    sections.append(f"📅 오늘 일정\n{_format_calendar_sections(data['calendar'])}")
else:
    sections.append("📅 오늘 일정\n  (없음)")
```

`data.get("calendar_ok", True)` 로 읽어 이전 XCom payload와 호환을 유지한다.

**4-4.** Step B LLM 프롬프트의 `cal_text` (약 1018행) 도 실패 시 `"  (수집 실패)"` 를 넘겨 LLM이 "일정 없음"으로 오해하지 않게 한다:

```python
if not data.get("calendar_ok", True):
    cal_text = "  (수집 실패)"
else:
    cal_text = "\n".join(f"  {e['time']} {e['summary']}" for e in data["calendar"]) or "  (없음)"
```

```python
import json
import logging
import multiprocessing
import os
import queue
import re
from datetime import datetime, date, timedelta
from pathlib import Path

import pendulum

logger = logging.getLogger(__name__)

_GIT_ROOT = Path(__file__).resolve().parent.parent.parent.parent.parent

_FLOW_CALENDAR_COLLECTION_TIMEOUT_SEC = 600
_FLOW_CALENDAR_PROCESS_TIMEOUT_SEC = int(os.getenv("FLOW_CALENDAR_PROCESS_TIMEOUT_SEC", "120"))
_FLOW_CALENDAR_NAV_TIMEOUT_SEC = 45
_FLOW_TEAM_LOGIN_WAIT_SEC = 15


def _collect_calendar_events_worker(result_queue) -> None:
    try:
        result_queue.put({"ok": True, "events": _collect_calendar_events_inner()})
    except BaseException as e:
        result_queue.put({"ok": False, "error": repr(e)})


def _collect_calendar_events() -> list[dict]:
    """Flow Calendar 수집을 격리한다. 멈추거나 실패하면 빈 목록으로 브리핑을 계속한다."""
    ctx = multiprocessing.get_context("fork" if hasattr(os, "fork") else "spawn")
    result_queue = ctx.Queue(maxsize=1)
    process = ctx.Process(target=_collect_calendar_events_worker, args=(result_queue,))
    process.start()
    process.join(_FLOW_CALENDAR_PROCESS_TIMEOUT_SEC)

    if process.is_alive():
        logger.warning("Flow Calendar 수집 프로세스 시간 초과(%s초) - 일정 수집 건너뜀",
                       _FLOW_CALENDAR_PROCESS_TIMEOUT_SEC)
        process.terminate()
        process.join(5)
        if process.is_alive():
            try:
                process.kill()
            except AttributeError:
                pass
            process.join(5)
        return []

    try:
        payload = result_queue.get_nowait()
    except queue.Empty:
        if process.exitcode not in (0, None):
            logger.warning("Flow Calendar 수집 프로세스 비정상 종료: exitcode=%s", process.exitcode)
        return []

    if not payload.get("ok"):
        logger.warning("Flow Calendar 수집 프로세스 실패: %s", payload.get("error", "unknown"))
        return []

    events = payload.get("events") or []
    return events if isinstance(events, list) else []


def _collect_calendar_events_inner() -> list[dict]:
    """Flow Calendar 오늘 일정 수집. 실패하면 빈 목록 반환."""
    driver = None
    try:
        ...
        deadline = time.monotonic() + _FLOW_CALENDAR_COLLECTION_TIMEOUT_SEC
        target_date = pendulum.now("Asia/Seoul").to_date_string()
        driver = _launch_flow_calendar_browser()
        _configure_flow_driver_timeouts(driver)   # ← BUG: 로그인 전에 45초 캡
        driver.set_window_size(1920, 1080)
        if not do_login(driver):
            logger.warning("Flow 로그인 실패 — 일정 수집 건너뜀")
            return []
        ...
    finally:
        if driver:
            try:
                driver.quit()
            except Exception:
                pass


def _configure_flow_driver_timeouts(driver) -> None:
    try:
        driver.set_page_load_timeout(_FLOW_CALENDAR_NAV_TIMEOUT_SEC)
        driver.set_script_timeout(_FLOW_CALENDAR_NAV_TIMEOUT_SEC)
    except Exception as e:
        logger.info("Flow Calendar WebDriver timeout 설정 실패(무시): %s", e)


def collect_briefing_data(**context):
    """브리핑에 필요한 모든 데이터 수집 후 XCom 저장."""
    calendar = _collect_calendar_events()
    failures, log_warnings = _collect_previous_day_dag_results_v2()
    git = _collect_git_status()
    freshness = _collect_data_freshness()
    scheduled = _collect_scheduled_dags()

    payload = {
        "calendar": calendar,
        "failures": failures,
        "git": git,
        "freshness": freshness,
        "scheduled": scheduled[:10],
        "log_warnings": log_warnings,
    }
    context["ti"].xcom_push(key="briefing_data", value=json.dumps(payload, ensure_ascii=False))
    msg = f"수집 완료 — 일정 {len(calendar)}건 / 실패 {len(failures)}건 / 로그오류 DAG {len(log_warnings)}건"
    logger.info(msg)
    return msg


def generate_briefing(**context):
    ti = context["ti"]
    data = json.loads(ti.xcom_pull(task_ids="collect_briefing_data", key="briefing_data"))
    ...
    cal_text = "\n".join(f"  {e['time']} {e['summary']}" for e in data["calendar"]) or "  (없음)"
    ...
    sections = []
    sections.append(f"[AI 브리핑] {today}\n"
                    f"FAIL {fail_cnt} / WARN {warn_cnt} / 로그오류 {log_err_cnt} / 로그경고 {log_warn_msg_cnt}")
    sections.append(f"🎯 오늘 우선순위\n{priority_text}")

    # 일정
    if data["calendar"]:                         # ← BUG: 실패와 0건이 구분 안 됨
```

- **캘린더 추출/파싱 로직을 건드리지 말 것** (`_extract_flow_calendar_events_from_html`, `_dedupe_briefing_calendar_events`, `_format_calendar_sections`, `_collect_flow_calendar_events_from_week_view`). 성공한 실행에서 12건을 정상 추출했으므로 이미 정상 동작한다.
- `_collect_calendar_events()` 반환 타입을 `tuple[list[dict], bool]` 로 바꾸면 **호출부가 `collect_briefing_data()` 하나뿐**인지 확인하고 함께 고칠 것.
- XCom payload는 `data.get("calendar_ok", True)` 로 읽어 **이전 payload와 하위 호환**을 유지할 것.
- 수집 실패가 `collect_briefing_data` 태스크를 **실패시키면 안 된다.** 브리핑은 계속 발송되어야 한다 (기존 방어적 설계 유지).
