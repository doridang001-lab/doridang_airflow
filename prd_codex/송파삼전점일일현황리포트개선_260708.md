# 송파삼전점 일일 현황 리포트 — 프롬프트·데이터마트 개선

## Task
`DB_Daily_Corporate_Store_Report` DAG(월~금 08:50)의 송파삼전점 전일 매출 텔레그램 보고를 **일단위 트래킹용 간단 실적요약**으로 압축하고, 로컬 LLM(qwen2.5/gpt-oss)이 상황 판단을 못하는 문제를 **상황별 가이드 보강 + 단계별 프롬프트**로 개선한다. 수정은 파이프라인 단일 파일에서만 이뤄진다.

## 핵심 전제 (탐색으로 확인된 사실 — 반드시 유지)
- **텔레그램 메시지는 100% 템플릿 결정형**이다. `_polish_llm_result`가 LLM의 `today_action`/`telegram_message`를 덮어쓰고 `_format_telegram_message`로 재생성한다. LLM은 CSV `main_reason`(=`reason`)만 실질 반영.
- `send_telegram`(notifier.py)은 **parse_mode 없는 plain text** 전송 → `#`,`▶`는 그대로 렌더되고 마크다운 이스케이프 불필요. **18자 줄바꿈이 화면 폭을 그대로 제어**한다.
- 사용자 확정: 레이아웃 = 섹션형 압축 / 비교지표 = 기존 전주동요일·최근4주평균 **유지 + 최근5일 평균 추가** / `today_action` = **템플릿 고정 유지**(LLM은 `원인` 해석만).

## Implementation Steps

1. **홀 시간대 경계 조정 — 점심 07~15, 저녁 16~00**
   `_classify_hall_time_slots`에서 점심 하한 `6 → 7`.
   ```python
   df.loc[((raw_hour >= 7) & (raw_hour < 16)).fillna(False), "time_slot"] = "점심"
   df.loc[((raw_hour >= 16) & (raw_hour < 24)).fillna(False), "time_slot"] = "저녁"
   ```

2. **최근 5일 평균 흐름 추가**
   - `_rolling_4w_avg` 패턴을 복제한 `_rolling_5d_avg(ref_date)` 신규. 대상일 = `for days in (1, 2, 3, 4, 5)`, 데이터 있는 날만 평균.
   - `_build_metrics`에 `"compare_5d_avg": _compare(cur["전체"], avg_5d)` 추가 (기존 `compare_4w_avg`·`compare_prev_week` 유지).
   - `_llm_payload`에 `"최근5일평균대비": metrics["compare_5d_avg"]` 추가.

3. **`_wrap18(text)` 신규 헬퍼**
   공백 포함 18자 초과 시 어절(공백) 경계에서 줄바꿈, 한 어절이 18자 초과면 하드 브레이크. 반환은 `\n`으로 join된 문자열. (한글 1글자=1로 계산 = `len()`)

4. **텔레그램 메시지 압축 재작성 — `_format_telegram_message(metrics, reason, actions)`**
   확정 레이아웃(숫자는 원 단위 유지, 본문 줄은 `_wrap18` 적용, 헤더 라인은 예외로 한 줄 유지):
   ```
   # 송파삼전점 직영점 일일 현황 요약

   ▶ 07/06 실적
   매출 1,777,263원
   (전일 -58.0%)
   주문 69건 (-54%)
   객단가 25,757원

   ▶ 최근5일 평균 대비
   매출 -25% / 주문 -20%

   ▶ 원인
   배달 매출 급감이
   가장 큼 (-230만원)

   ▶ 오늘 할 일
   11시 전 배민·쿠팡
   영업/품절 확인
   ```
   - 날짜 헤더는 `MM/DD` 포맷(`metrics["report_date"]`에서 파생).
   - `▶ 원인`: 인자로 받은 `reason`(LLM값 우선) 사용.
   - `▶ 오늘 할 일`: `actions`(= `_operation_action_candidates(metrics)[:2]`)를 18자 줄바꿈해 나열.
   - 기존 `## 원인별 실행 가이드` 블록 및 `_guide_lines` 호출 **제거**(9줄짜리 장문 삭제).

5. **`_polish_llm_result` 조정**
   - `today_action`은 템플릿 생성 유지.
   - `reason`은 LLM 값 보존(단 `_is_generic_llm_result` 필터 통과분만, 실패 시 `_structured_reason_text` fallback). 그 `reason`을 `_format_telegram_message`에 전달.

6. **상황별 가이드 대폭 보강 — `SITUATION_GUIDELINES` + `_applied_situations`**
   로컬 LLM이 숫자→상황 판단을 못하므로 **판정 임계값을 조건 문자열에 명시**하고 상황 확장.
   - 기존 6개(`배달_주문감소`,`홀_주문감소`,`객단가_하락`,`재구매_약함`,`매출_증가`,`데이터_미준비`) 유지 + `condition`에 수치 명시(예: "배달 orders_pct <= -20").
   - 신규 추가 예: `주문수_급감`(orders_pct<=-30), `홀_점심약세`/`홀_저녁약세`(time_slots 점심/저녁 매출 판정), `신규고객_부진`(new_ratio<=20), `배달비중_과다`, `전주比_하락`(compare_prev_week.sales_diff<0), `5일평균_하회`(compare_5d_avg.sales_diff<0).
   - `_applied_situations`에 신규 상황 판정 로직 추가 → payload `적용된_상황`으로 LLM에 전달.

7. **LLM 프롬프트 개선 — `_llm_prompt`**
   - 목적 문구에 **"일단위 트래킹"** 명시.
   - **단계별 판단 가이드** 추가: (a) 매출 방향 → (b) 홀/배달 중 기여 큰 채널 → (c) 주문수 vs 객단가 원인 → (d) 해당 상황 가이드 조치 선택. "제공된 `핵심판정`·`적용된_상황`·`상황별_가이드라인`을 근거로만 쓰라" 지시.
   - `reason` 제약: 45자 이내 1문장 + 숫자 근거 1개 포함.
   - 좋은 예/나쁜 예를 최근5일·전일 기반으로 갱신.

8. **마크다운 리포트 `_markdown_report`**
   - `## 최근5일 평균 대비`(매출/주문/객단가 증감) 섹션 추가, 간결 유지.

## Constraints
- **단일 파일만 수정**. DAG·schedule·notifier·paths 변경 금지.
- 텔레그램 메시지는 **plain text**(parse_mode 없음) — 백틱/마크다운 강조 문법 넣지 말 것(그대로 노출됨).
- 숫자는 **제공된 metrics 값만** 사용, 임의 계산·추정 금지. 금액 포맷은 기존 `_fmt_won`/`_fmt_int`/`_diff_text`/`_pct` 재사용.
- `today_action`은 템플릿(`_operation_action_candidates`) 유지 — LLM 생성으로 바꾸지 말 것.
- 기존 비교지표(`compare_prev_week`, `compare_4w_avg`)는 **삭제하지 말고 유지**, 5일평균은 추가.
- 본문 줄바꿈은 반드시 `_wrap18` 경유(헤더 `#`, 섹션 헤더 `▶ ...` 라인 제외 선택 가능하나 결과 줄이 18자 넘지 않게).
- print 금지, `logger` 사용.
