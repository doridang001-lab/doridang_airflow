# 쿠팡 멀티매장(상도점 외 2개) 수집 안정화

## Task
쿠팡 확장 배치 수집에서 **한 로그인에 매장이 여러 개 묶인 멀티매장 계정**(`schoolfoodseoul` 3매장, `doriys` 2매장)만 계속 실패한다. 증상은 "1번째 매장만 수집하고 2번째 매장부터 광고(CMG)로 넘어감 → `orders 타임아웃 — 0/0개`". 원인은 멀티매장 orders 루프가 **단일 JS 실행**으로 드롭다운 매장전환을 처리하는데, Wing의 매장전환이 **페이지 reload를 유발**해 실행 중이던 content-script 루프가 파괴되기 때문. 기존 `ce_orders_checkpoint` + `ce_orders_restart` 인프라를 배치 경로에 연결해 **reload를 사이에 두고 매장을 이어달리도록** 고친다.

## Implementation Steps

### Step 0 — 실측 확인 (구현 전 필수)
확장 로드된 크롬에서 **doriys**(2매장) 만 "선택 테스트"로 실행하고 작업 탭 DevTools 콘솔/Network를 관찰한다.
- 1번째 매장 수집 후 `_selectStoreInOrdersDropdown` 로그(`→ 매장 선택 시도`/`✅ 매장 선택 완료`) 직후 **페이지 document 재요청(reload) 또는 URL 변경**이 일어나는지 확인.
- **reload 확정** → Step 1 진행.
- reload 아님(SPA인데 2번째 매장 앵커 미탐지/조회 실패) → `_selectStoreInOrdersDropdown`/`_getOrdersDropdownStores`의 셀렉터·드롭다운 재오픈 타이밍 보강으로 방향 전환.

### Step 1 — 매장 전환 직전 재개 상태 저장 (1a)
`_collectOrdersMultiStore` 루프에서, 다음 매장으로 넘어가기 위해 `_selectStoreInOrdersDropdown(nextStore)` 로 **전환 클릭을 하기 직전에** (배치 모드 && 남은 미완료 매장 존재 시):
1. `chrome.storage.local.set` 로 저장:
   - `ce_orders_restart = { date: <today YYYYMMDD> }`
   - `ce_current_target_stores`(배열), `ce_current_target_date`, `ce_current_target_date_mode` (opts 그대로)
2. `ce_orders_batch_reload_count` 증가 (신규 키, 또는 checkpoint 필드). `MAX_STORE_RELOAD = matchingStores.length + 2` 초과 시 → 무한 reload 방지: 남은 매장을 `blocked/부분완료`로 표시하고 즉시 완료신호(1c) 후 종료.
3. 이미 완료한 매장은 이 시점에 `ce_orders_checkpoint`(2640)에 저장돼 있어야 함 → reload 후 스킵됨.

> 핵심: "전환 클릭 → reload → 05_main load 핸들러가 배치 재개 → checkpoint로 완료매장 스킵 → 다음 미완료 매장 수집" 사이클.

### Step 2 — reload 후 자동 재개 경로 (1b)
`05_main.js:60-68` 의 `ce_orders_restart` 핸들러는 이미 `runCollector({ source:'batch', targetStores, targetDate, targetDateMode })` 를 호출한다. 따라서 **Step 1에서 restart 키를 실제로 심기만 하면** 재개가 동작한다.
- 재진입한 `_collectOrdersMultiStore` → `_loadOrdersCheckpoint()`(2486) → 완료매장 스킵(2518) → 다음 매장 수집 → 또 전환 시 Step 1 반복.
- 재개 조건이 오늘 날짜(`resume.date === today`)와 일치해야 함(기존 로직).

### Step 3 — 완료신호(MULTISTORE_COMPLETE) 정확히 1회 (1c)
현재 코드는 루프 끝(2654-2668)에서 **매 run 무조건** `MULTISTORE_COMPLETE` 를 보낸다 → 중간 run(1매장만 하고 reload로 넘어가는 경우)에서 조기 발신되어 runner가 오판할 수 있음. 다음처럼 가드:
1. **모든 `matchingStores` 가 완료(checkpoint/storeResults 에 completed/blocked/hardThrottle)** 일 때만 `MULTISTORE_COMPLETE` 발신 + `_clearOrdersCheckpoint()` + `ce_orders_restart`/target/`batch_reload_count` 정리.
2. 아직 남은 매장이 있으면 → 발신하지 않고 Step 1(restart 저장)로 reload 유도 후 `return`.
3. `MAX_STORE_RELOAD` 한도 도달 시 → 남은 매장 부분완료 처리하고 **즉시** 완료신호 발신(600초 낭비 제거).

### Step 4 — 회귀 방지 (Step 2 in plan)
- 단일매장: `_getOrdersDropdownStores`가 `.dropdown-btn.highlight` 없으면 `[]` → 단일매장 경로(2401-2478) 그대로. 재개 로직은 `matchingStores.length > 1` && 배치일 때만 활성.
- manual 모드: 재개 로직은 `this._isDashboardBatchOrders(opts)` 참일 때만 (기존 manual F5 경로와 분리 유지).
- `doriys`(2)·`schoolfoodseoul`(3) 모두 동일 경로 → 계정 분기 불필요.

### content/03_coupangeats.js — 매장 선택(전환) 함수
```javascript
async _selectStoreInOrdersDropdown(storeText) {
  const toggleBtn = document.querySelector('.dropdown-btn.highlight');
  if (!toggleBtn) return false;
  toggleBtn.click();
  let list = document.querySelector('.dropdown-list');
  if (!list || list.offsetParent === null) {
    for (let i = 0; i < 20; i++) {
      await new Promise(r => setTimeout(r, 50));
      list = document.querySelector('.dropdown-list');
      if (list && list.offsetParent !== null) break;
    }
  }
  const items = [...document.querySelectorAll('.dropdown-list li a')];
  const targetKey = this._normalizeStoreName(storeText);
  const target = items.find(el => {
    const t = el.textContent.trim();
    const key = this._normalizeStoreName(t);
    return key === targetKey || key.includes(targetKey) || targetKey.includes(key);
  });
  if (!target) { toggleBtn.click(); return false; }
  target.click();               // ← 이 클릭이 페이지 reload를 유발(추정) → 루프 파괴
  for (let i = 0; i < 10; i++) {
    await new Promise(r => setTimeout(r, 50));
    const closedList = document.querySelector('.dropdown-list');
    if (!closedList || closedList.offsetParent === null) break;
  }
  return true;
}
```

### content/03_coupangeats.js — 체크포인트 헬퍼 + 저장 시점
```javascript
_loadOrdersCheckpoint() {
  return new Promise((resolve) => {
    chrome.storage.local.get(['ce_orders_checkpoint'], (r) => resolve(r.ce_orders_checkpoint || null));
  });
}
_saveOrdersCheckpoint(data) {
  return new Promise((resolve) => chrome.storage.local.set({ ce_orders_checkpoint: data }, resolve));
}
_clearOrdersCheckpoint() {
  return new Promise((resolve) => chrome.storage.local.remove(['ce_orders_checkpoint'], resolve));
}
```

## Constraints
- `runner.js`의 `COLLECT_TIMEOUT/CMG_TIMEOUT/MENU_TIMEOUT` 값 자체는 변경 금지(안전망 유지).
- `accounts.js` 변경 금지. 구버전 파일(루트 `content.js`, `*-DESKTOP-*`, `*.bak_*`) 수정 금지.
- 완료신호(`MULTISTORE_COMPLETE`)는 **전 매장 완료 시 정확히 1회만** — 중간 run 조기 발신 금지.
- 매장전환 reload 재개는 **배치 모드(`_isDashboardBatchOrders`)** && `matchingStores.length > 1` 일 때만. manual/단일매장 경로 무영향.
- 무한 reload 방지 위해 `MAX_STORE_RELOAD` 한도 필수.
- 매장명 매칭은 반드시 `_normalizeStoreName` 경유.
