# 쿠팡 orders 수집: 0건 매장에서 10057 오탐 제거

## Task

쿠팡이츠 크롬 확장의 배치 orders 수집에서, 주문이 0건인 매장이 `10057: 과도한 요청` 으로 오판되어 "차단됨"으로 기록되고 다음 날 백필 큐로 이월된다. 실제로는 제한이 아니라 `<div class="order-search-notice">조회할 내역이 없습니다</div>` 만 뜬 정상 0건 상태다. 오판 → 불필요한 재조회 → 실제 쿠팡 제한 유발이라는 악순환이 생긴다.

10057 감지를 정밀화하고, 빈 결과 공지를 0건의 근거로 삼아 재조회 없이 다음 매장으로 넘어가게 한다.

### 근거 (실제 배치 로그)

```
03:23:12  🔍 조회 버튼 클릭 (1차)
03:23:13  ✅ 조회 버튼 클릭 성공
03:23:13  ⏳ 데이터 로드 대기 중 (0건 → 변경 대기)
03:23:13  ⛔ 쿠팡 제한(10057) 감지 — 현재 매장 패스   ← 클릭 1초 후
03:23:13  ⛔ 쿠팡 제한(10057) 감지 — 현재 매장 패스   ← 같은 초에 3번
03:23:13  ✅ 데이터 로드 완료 (0건)
03:23:13  ⛔ 쿠팡 제한(10057) 감지 — 현재 매장 패스
```

조회 클릭 1초 후 같은 초에 3번 감지됐다. 실제 제한 팝업이라면 불가능하다. 3번은 각각 `_waitForOrdersCountChange`, `_waitForOrdersDataReady`, 멀티스토어 루프에서 호출한 `_checkOrdersHardThrottle()` 이다. 매번 true를 반환했다는 것은 **매칭된 문자열이 응답이 아니라 페이지에 상시 존재하는 정적 텍스트**라는 뜻이다.

범인은 `_isOrdersThrottleVisible()` 의 `document.body.innerText.includes('10057')` 맨숫자 부분매치다. 매장번호·사업자번호 등 무관한 숫자에 `10057` 다섯 글자가 포함되면 무조건 제한으로 오판한다. 특정 계정(`schoolfoodseoul`, `vloolv90`)에서만 재현되고 사용자가 직접 열면 멀쩡한 점이 이 가설과 맞는다.

두 번째 원인: `_isNormalZeroOrders()` 는 공지가 떠 있어도 `!this._isOrdersThrottleVisible()`(위 오탐에 오염) 와 `!this._hadRecentOrdersTransientError()`(직전 매장 에러가 120초간 잔류) 때문에 false가 된다. 60개 매장 배치에서 앞 매장의 에러가 뒤 매장의 정상 0건 판정을 막는다.

### Step 6 (먼저 적용) — 이전 매장의 잔존 공지를 새 결과로 오인하지 않게 가드

**왜 먼저인가:** Step 3을 그대로 적용하면 새로운 매출 누락 버그가 생긴다. 매장 전환은 SPA 갱신이라, 조회 클릭 직후 응답이 오기 전까지 **직전 매장의 `.order-search-notice` 가 DOM에 남아 있을 수 있다.** 직전 매장이 0건이면, 새 매장 결과가 도착하기도 전에 공지를 보고 "0건"으로 스킵해버린다.

로그에 조짐이 있다. 상도점 시작 시 `prevCount = 46건`(이전 매장 잔존값), 양천목동점 시작 시 `prevCount = 0건`(상도점의 0건 잔존값). `_waitForOrdersCountChange` 는 주석대로 `prevCount === 0` 이면 0→0 변화를 감지할 수 없어 3초만 기다린다.

1. `_markExistingNoticesStale()` 헬퍼를 추가한다. 현재 DOM의 `.order-search-notice` 노드 전부에 `data-collector-stale="1"` 속성을 찍는다.
2. `_clickOrdersSearchButton()` (아래 Reference Code 참고) 안에서, `_waitBeforeOrdersSearch()` 통과 직후 **버튼을 클릭하기 전에** `this._markExistingNoticesStale()` 를 호출한다. 성공 반환하는 두 경로(셀렉터 매치 / SVG path 폴백) 모두에 앞서 한 번만 호출되면 된다.
3. `_hasNoOrdersNotice()` 의 셀렉터를 `.order-search-notice:not([data-collector-stale])` 로 바꾼다.

SPA가 리렌더하면 속성 없는 새 노드가 생기므로 자연히 통과한다. 리렌더가 없으면 공지는 무시되고 기존 카운트/요약 기반 판정으로 넘어간다.

### Step 1 — 10057 감지 정밀화 (`_isOrdersThrottleVisible`)

`_findOrdersThrottleSignal()` 헬퍼를 새로 만들고, `_isOrdersThrottleVisible()` 는 그 결과의 boolean 변환으로 남긴다. 헬퍼는 `{ matched: boolean, keyword: string, snippet: string }` 을 반환한다 (`matched: false` 면 나머지는 빈 문자열).

판정 순서:

1. **에러 컨테이너 스코프 검사 (1순위).** `ERROR_CONTAINER_SELECTORS` 상수를 파일 상단 근처에 정의하고, `_closeErrorPopups()` 가 이미 쓰는 목록을 재사용한다:
   ```js
   ERROR_CONTAINER_SELECTORS: ['.modal.show', '.ant-modal', '[role="dialog"]', '.popup-wrapper', '.toast', '[role="alert"]'],
   ```
   이 컨테이너들의 `textContent` 안에서 `10056` / `10057` / `ACCESS_DENIDE` / `ACCESS_DENIED` / `ACCESS_DENY` / `과도한 요청` / `일시적으로 제한` / `단시간 내 반복 요청` 중 하나라도 매치하면 `matched: true`.
2. **공지 우선 (2순위 진입 전 차단).** 1순위에서 매치가 없고 `this._hasNoOrdersNotice()` 가 true면 즉시 `matched: false` 로 반환한다. 정상 빈 결과 화면은 절대 throttle이 아니다.
3. **body 폴백은 문맥을 요구 (3순위).** `document.body.innerText` 를 쓰되 맨숫자 부분매치를 버린다:
   - 숫자 코드: `/(?:^|\D)1005[67](?:\D|$)/` 로 경계를 잡고, **그 매치 위치 앞뒤 80자 안에 `제한` 또는 `과도한 요청` 이 함께 있을 때만** 인정한다.
   - 문자열 키워드: `일시적으로 제한`, `서비스 이용이 일시적`, `단시간 내 반복 요청`, `과도한 요청`, `ACCESS_DENIDE`, `ACCESS_DENIED`, `ACCESS_DENY` 는 그대로 인정.
   - **`'과도한 요'` 는 `'과도한 요청'` 으로 교정한다.** 기존의 짧은 조각은 삭제.
4. `snippet` 은 매치 지점 앞뒤 30자씩(총 60자) 잘라 담는다.

기존 호출부(`_checkOrdersHardThrottle`, `_isNormalZeroOrders`, `_collectOrdersPages` 등)는 `_isOrdersThrottleVisible()` 를 계속 쓰므로 시그니처를 바꾸지 않는다.

### Step 2 — 감지 근거 로깅 + 중복 마킹 방지 (`_markOrdersHardThrottle`)

1. `_markOrdersHardThrottle()` 진입부에서 **이미 `this._hardThrottled === true` 면 즉시 return** 한다. 카운터도 올리지 않고 로그도 남기지 않는다. 로그에 같은 초에 3번 찍힌 게 이 때문이고, `_ordersThrottleCount` 가 부풀면 `_ordersThrottleDelayMs()` 가 불필요하게 210초까지 올라간다.
2. `_findOrdersThrottleSignal()` 의 `keyword` / `snippet` 을 debug 로그에 함께 남긴다:
   ```js
   Utils.updateProgressModal({ debug: `⛔ 쿠팡 제한(10057) 감지 — 현재 매장 패스 | keyword=${signal.keyword} | ...${signal.snippet}...` });
   ```
   `_checkOrdersHardThrottle()` 이 signal 객체를 받아 `_markOrdersHardThrottle(signal)` 로 넘기도록 고친다.

`runner.js:1422` 의 `issueKeywords` 필터가 `10057` 을 포함하므로 이 로그는 문제 로그 md에 자동 수집된다.

### Step 3 — 공지를 0건의 최종 근거로 (`_isNormalZeroOrders`)

`_hasNoOrdersNotice()` 가 true이고(= Step 6에 의해 **stale이 아닌 공지**) 에러 컨테이너에 제한 팝업이 없으면 **즉시 true를 반환**한다. `_getExpectedOrderCount()` / `_getExpectedSalesTotal()` 의 잔존 요약값이나 `_hadRecentOrdersTransientError()` 의 120초 잔류가 판정을 뒤집지 못하게 한다.

`requireNotice: false` 로 호출하는 기존 경로의 의미는 그대로 둔다 (공지 없이 카운트/매출 0 + throttle 없음 + transient error 없음).

0건이 확정되면 아래를 리셋해 직전 매장의 에러 상태가 뒤 매장으로 새지 않게 한다:
```js
this._lastOrdersTransientErrorAt = 0;
this._ordersThrottleCount = 0;
```

### Step 4 — 대기 단축 및 판정 순서 교정

1. `_waitForOrdersDataReady()` 폴링 루프 안에서 `this._hasNoOrdersNotice()` 가 참이면 즉시 `return true`. 현재는 0건일 때 8초 타임아웃을 꽉 채우고, `_waitForOrdersCountChange` 의 3초까지 합쳐 매장당 ~11초를 버린다.
2. `_collectOrdersMultiStore()` 의 매장 루프에서 **`_checkOrdersHardThrottle()` 보다 먼저** 0건 공지를 확인한다. 현재 순서(throttle 체크 → 0건 체크)면 오탐이 0건을 가로챈다. 아래 Reference Code의 `2531` / `2537` 블록 순서를 뒤집는다.
3. 같은 블록의 `storeResults.push(...)` 에 `zeroOrders: true` 를 추가한다.
   `runner.js` 의 `categoryPayloadOk()` 가 이미 `zeroOrders` 를 성공으로 인정하고, `completed: true` 이므로 `incompleteStores()` 에도 걸리지 않는다 → 재시도 라운드와 다음 날 백필 큐에서 자동으로 빠진다.

### Step 5 — 0건 매장 카드가 "대기 중..."에 멈추는 버그

사용자가 "양천목동점을 클릭했는데 상도점을 수집 중인 것으로 보인다"고 관찰한 현상의 원인. **데이터가 섞이는 게 아니라 UI 상태 갱신 누락이다.**

`Utils.setMultiStoreIndex(i)` 는 카드를 `⏳` 로만 바꾸고, 최종 상태는 `Utils.updateMultiStoreStoreResult()` 가 써야 한다. 그런데 0건 분기는 `storeResults.push()` 만 하고 이 함수를 호출하지 않는다. beacon 스킵 분기와 검증 경로(`_collectOrdersWithValidation`)는 모두 호출한다. 그래서 0건 매장 카드가 ⏳ "대기 중..." 으로 굳은 채 남고, 그 아래로 다음 매장 로그가 이어져 오해를 부른다.

수정: 0건 분기에 아래 한 줄 추가.
```js
Utils.updateMultiStoreStoreResult('success', '0건');
```

참고: `_selectStoreInOrdersDropdown` 의 이름 매칭은 `_normalizeStoreName` 적용 후 `도리당상도점` / `도리당양천목동점` 로 서로 `includes` 가 성립하지 않아 오선택 가능성은 없다. 매장 선택 로직은 건드리지 않는다.

구현 완료 후 **자동 실행 가능한 Test Case 1~3이 모두 PASS할 때까지** 반복한다.

LOOP until all PASS:
  1. Test Case 1~3 순서대로 실행
  2. FAIL 항목 → 원인 분석
  3. 코드 수정 (양쪽 폴더에 동일 반영)
  4. 전체 재실행
종료 조건: 1~3 전체 PASS + Constraints 위반 없음

Test Case 4~7은 브라우저가 필요하므로 실행하지 말고, 최종 보고에 **"사용자 수동 검증 필요"** 목록으로 그대로 옮겨 적는다.
