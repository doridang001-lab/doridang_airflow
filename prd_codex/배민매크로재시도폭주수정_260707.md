# 배민 매크로 재시도 폭주 수정

## Task
`DB_Beamin_Macro_Dags_Retry`(자기 자신을 재트리거하는 재시도 오케스트레이터)가 불가매장이 하나라도 있으면 폭주한다. 2026-07-01 로그에서 `attempt_10`(=`MAX_ATTEMPTS=10` 상한)까지 도달해 약 8시간 재시도, `attempt_2` run 폴더가 7개로 팬아웃, run_id가 부모 run_id를 통째로 중첩해 Windows 260자 경로 한도 초과가 발생했다. 매장별 재시도 ledger 도입·상한 축소·run_id 결정론화로 **불가매장이 있어도 빠르게 수렴**하고 **체인 증식/경로 폭주가 없도록** 만든다. 일일 수집 로직은 정상이므로 건드리지 않는다.

## Implementation Steps

### 1. 매장별 재시도 ledger (근본 수렴) — `DB_Beamin_retry.py`
- 파일 상단(상수 영역)에 추가: `PER_STORE_MAX_RETRY = 2` (한 매장은 재시도 루프에서 최대 2회만 재수집).
- store key 헬퍼: `def _store_key(account_id, store_id) -> str: return f"{str(account_id).strip()}::{str(store_id).strip()}"`.
- `build_retry_conf`(최초 트리거, 반환 dict): `"retry_history": {}` 키 추가.
- `build_next_retry_conf`:
  - 함수 시작부에서 `prev_history = dict(previous_conf.get("retry_history") or {})` 읽기.
  - 기존 로직으로 `failed_ads = _group_retry_stores(ad_items, "stores")`, `failed_orders = _group_retry_stores(toorder_items, "stores")` 산출한 **직후**, 아래 필터를 적용한다:
    - `new_history = dict(prev_history)`
    - 각 그룹(`failed_ads`, `failed_orders`)의 account 엔트리에서 `stores` 리스트를 순회하며
      `key = _store_key(account_id, store.get("store_id"))` 로 `prev_history.get(key, 0)` 조회.
      `>= PER_STORE_MAX_RETRY` 이면 해당 store 제외, 아니면 유지하고 `new_history[key] = prev + 1`.
    - `stores`가 비면 그 account 엔트리 자체를 드롭.
  - 필터 후 `failed_account_ids`를 필터된 `failed_ads`/`failed_orders` 기준으로 `_unique(...)` 재계산.
  - 반환 conf에 `"retry_history": new_history` 추가.
- 효과: 소진된 매장이 다음 conf에서 사라져 `next_target_count == 0` → `DB_Beamin_Macro_Dags_Retry.py`의 기존 종료 가드로 루프가 즉시 멈춘다.

### 2. 재시도 상한 10 → 3
- `dags/db/DB_Beamin_Macro_Dags_Retry.py`: `MAX_ATTEMPTS = 10` → `MAX_ATTEMPTS = 3`.
- `modules/transform/pipelines/db/DB_Beamin_retry.py`: `build_retry_conf(..., max_attempts: int = 10)` → `= 3`.
- `dags/db/DB_Beamin_Macro_Dags.py`: `int(conf.get("max_attempts", 10))` → `int(conf.get("max_attempts", 3))`.

### 3. run_id 결정론화 (중첩·팬아웃 제거) — `DB_Beamin_Macro_Dags_Retry.py` `notify_and_trigger_next`
- 기존(부모 run_id 중첩 + uuid):
  ```python
  parent = _safe_run_id_part(str(context.get("run_id") or "manual"))
  run_id = f"retry__{str(target_date).replace('-', '')}__attempt_{attempt + 1}__{parent}__{uuid.uuid4().hex[:8]}"
  ```
- 변경(안정 루트 = 원본 스케줄 run_id, uuid 제거, 중첩 제거):
  ```python
  root = _safe_run_id_part(str(conf.get("source_run_id") or context.get("run_id") or "manual"))
  run_id = f"retry__{str(target_date).replace('-', '')}__attempt_{attempt + 1}__{root}"
  ```
- `import uuid`가 파일 내 다른 곳에서 안 쓰이면 제거.
- 결과: (target_date, attempt, source_run_id)에 대해 결정론적 → `DagRunAlreadyExists`가 중복 재트리거 차단, 폴더명 길이 상수화.

## Constraints
- `retry_needed` 로직은 변경하지 않는다 (ledger가 대상을 비우면 `next_target_count==0`로 수렴).
- 일일 수집 로직(NOW/우가/변경이력/orders/ad_funnel), 렌더러 타임아웃 분류, "운영시간 요소 로드 실패" 스왈로우 처리는 건드리지 않는다.
- `build_retry_conf`/`build_next_retry_conf` 반환 dict의 기존 키를 제거하지 말 것 (conf 하위호환). `retry_history`는 **추가만**.
- 최초 트리거(`trigger_retry_if_needed`)의 run_id는 이미 결정론적이므로 유지.
- `retry_history` 키가 없는 옛 conf가 들어와도 동작해야 함 → `previous_conf.get("retry_history") or {}` 로 방어.
