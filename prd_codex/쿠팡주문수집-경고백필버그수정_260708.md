# 쿠팡 주문 수집 — "경고(throttle) → 무한 재도전" 버그 수정

## Task
쿠팡이츠 주문 자동 수집(batch)에서 매장이 **soft/silent 경고(과도한 요청·반복요청 제한·상세패널 무음 차단)**를 먹으면, 지금은 `status:'fail'`로 처리돼 runner의 **당일 자동 재시도 루프(최대 5회, 5~10분 간격)**에 다시 들어가 같은 매장을 계속 재조회한다. 이 재조회가 오히려 10057 영구차단을 유발한다. 경고성 미완료는 **`blocked:true`(비고만 저장 → 다음날 백필로 이월)**로 분기해 당일 재시도 루프 진입 자체를 차단한다.

## 근본 원인
`_collectOrdersMultiStore`의 배치 멀티매장 루프(2472-2476)에서 `collected===false`이면 **실패 이유를 안 보고** 무조건 `status:'fail', error:true`로 push한다.
- 명시적 10057(`_hardThrottled`)은 바로 위 2468-2471에서 `_pushOrdersBlockedCurrentStore()`로 `blocked:true` push → runner가 백필로 이월 (정상).
- 그러나 silent throttle은 `_collectOrdersWithValidation`가 `this._lastOrdersCollectResult = { partialThrottle:true, detailBlocked, note:'부분 저장 후 백필' }`(03_coupangeats.js:3766)까지 세팅해 **"백필로 넘겨라"는 의도**를 남기는데, 호출부 2472는 이걸 **무시**하고 boolean만 보고 `fail` 처리한다.
- runner 흐름: `fail`(blocked=false, ordersOk=false) → `incompleteStores()`(runner.js:312) 포함 → `scheduleAutoRetry`(runner.js:507) → `runRetry`(runner.js:1287) 최대 5회. `blocked` 매장만 재시도에서 제외된다(runner.js:314).

## Implementation Steps

### 1) 2472-2476 블록을 throttle/경고 여부로 분기
`collected===false`일 때 `this._lastOrdersCollectResult`와 throttle 상태를 확인해:
- **경고/불안정** → `blocked:true` 결과로 push + 비고 저장 + 다음 매장 전 쿨다운 → runner가 백필로 이월(당일 재시도 X)
- **진짜 실패**(매장선택/조회 실패 등, throttle 아님) → 기존대로 `fail`(재시도 대상)

기존 헬퍼 `_makeOrdersBlockedResult()` / `_waitAfterOrdersBlockedStore()`(03_coupangeats.js:2166, 2120)를 재사용한다.

**변경 전 (2472-2476):**

        storeResults.push({ storeName, completed: false, status: this._stopFlag ? 'warn' : 'fail', error: true, note: this._stopFlag ? '중단됨' : '수집 결과 없음' });

**변경 후:**
```javascript
      if (!collected) {
        const r = this._lastOrdersCollectResult || {};
        const throttled = !this._stopFlag && (

        continue;
      }
```

## Test Cases
확장 로직이라 실제 쿠팡 재현이 어려우므로 **경로 추적 + devtools 강제 주입** 중심으로 검증한다.

1. [문법 검증] 수정 후 파일이 파싱 오류 없이 로드되는지 — chrome://extensions 에서 확장 재로드 → 에러 배지 없음.
2. [경고→백필] runner에서 임의 매장 배치 실행 중, devtools 콘솔에서 `document.body.insertAdjacentHTML('beforeend','<div>과도한 요청</div>')`로 경고 강제 → 해당 매장 로그에 **"백필 이월"** 뜨고, 배치 종료 후 retryBtn 자동재시도 예약이 **안 뜨는지**(incompleteStores=0) 확인.
3. [백필 큐 저장] 위 2번 후 `chrome.storage.local.get('ce_backfill_queue')` → 해당 매장이 `failedStores`에 포함.
4. [진짜 실패 대조] '매장 선택 실패'를 강제(존재하지 않는 매장 지정)하면 기존대로 **fail → 자동재시도 예약**이 뜨는지 대조.

## Verification Loop
```
LOOP until all PASS:
  1. Test Cases 1→4 순서대로 실행
  2. FAIL 항목 → 원인 분석 (특히 throttled 분기 조건이 너무 넓/좁은지)
  3. 코드 수정
  4. 전체 Test Cases 재실행
종료 조건: 전체 PASS (경고=백필, 진짜실패=재시도 구분 정확)
```
